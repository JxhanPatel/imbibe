# 788. Rotated Digits 🟡

**Topics:** Math, Digit Processing, Enumeration






## Intuition

Think of each number like a digital display on a calculator. When you rotate it upside down, some digits still make sense, some change, and some break completely.

* Digits like `0, 1, 8` are stable, they look the same.
* Digits like `2, 5, 6, 9` transform into different digits.
* Digits like `3, 4, 7` become invalid and break the number.

A number is **good** if:

1. Every digit is valid after rotation.
2. At least one digit changes (so the number is different).

So the task becomes:
For every number from `1` to `n`, check if it satisfies these rules.








## Approach

We will iterate through each number from `1` to `n` and check:

1. Convert number to string to inspect digits.
2. For each digit:

   * If it's invalid (`3, 4, 7`), reject immediately.
   * If it's a changing digit (`2, 5, 6, 9`), mark that the number changes.
3. After checking all digits:

   * Count it only if:

     * It has no invalid digits
     * It has at least one changing digit








## Example Walkthrough

### Input: `n = 10`

We check each number:

* `1` → valid but no change → skip
* `2` → valid and changes → count
* `3` → invalid → skip
* `4` → invalid → skip
* `5` → valid and changes → count
* `6` → valid and changes → count
* `7` → invalid → skip
* `8` → valid but no change → skip
* `9` → valid and changes → count
* `10` → digits `1,0` → valid but no change → skip

Final count = `4`








## Code

```python
class Solution:
    def rotatedDigits(self, n: int) -> int:

        bad = set('347')
        must_change= set('2569')

        count = 0

        for num in range(1, n+1):
            digits = str(num)

            if any(d in bad for d in digits):
                continue

            if any(d in must_change for d in digits):
                count +=1


        return count
```







## Complexity

* Time: $O(n \cdot d)$ where $d$ is number of digits (at most 5 here)
* Space: $O(1)$






## Edge Cases

* `n = 1` → no good numbers
* All digits are from `{0,1,8}` → none are good
* Numbers containing `3,4,7` → immediately invalid
* Large `n` near `10^4` still works efficiently





## Lessons

* Treat digits independently when transformation rules are per-digit
* Separate conditions into:

  * validity check
  * transformation check
* Early rejection (invalid digits) saves time
* This pattern is common in digit-based problems like "confusing number" or "strobogrammatic number"
