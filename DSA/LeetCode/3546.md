# 3546. Equal Sum Grid Partition I 🟡

**Topics:** Array, Matrix, Prefix Sum


## Intuition

Think of it like splitting a chocolate bar. A chocolate bar has rows of squares, and you want to snap it along one line so both halves have the same total number of chocolate squares. You can only snap it horizontally (across a row boundary) or vertically (along a column boundary), and both halves must be non-empty.

The key insight is: if the total sum of the whole bar is even, there might exist a snap point where the left/top half equals exactly `total / 2`. If the total is odd, it is literally impossible since integers cannot split into two equal halves.

## Approach

We use a classic **prefix sum** technique. The idea is to scan row by row (or column by column) and keep a running total. At each potential cut point, we check if the running sum equals exactly half the total.

Here is the plan step by step:

1. Compute the total sum of the entire grid.
2. If `total` is odd, immediately return `False` since two equal integer parts are impossible.
3. Scan rows from top to bottom, accumulating a `prefix` sum. After each row (except the last, since both parts must be non-empty), check if `prefix == total // 2`. If yes, a valid horizontal cut exists.
4. Do the same scan column by column for a vertical cut.
5. If neither scan finds a valid cut, return `False`.

This avoids recalculating sums from scratch at every cut position, keeping things efficient.

## Example Walkthrough

Using `grid = [[1, 4], [2, 3]]`:

- Total sum = 1 + 4 + 2 + 3 = **10**, and `10 // 2 = 5`
- **Horizontal scan:**
  - After row 0: prefix = 1 + 4 = **5**. Is 5 == 5? Yes! And it's not the last row. Valid cut found.
- Return `True` immediately.

For the vertical scan on `grid = [[1, 3], [2, 4]]` (total = 10, target = 5):
- After column 0: prefix = 1 + 2 = **3**. Is 3 == 5? No.
- That was the only possible cut (column 1 would be the last column, leaving nothing on the right).
- No valid vertical cut.

## Code

```python
class Solution:
    def canPartitionGrid(self, grid: List[List[int]]) -> bool:

        m, n = len(grid), len(grid[0])

        total = sum(sum(row) for row in grid)

        if total % 2 != 0:
            return False

        target = total // 2

        prefix = 0
        for i in range(m - 1):     

            prefix += sum(grid[i])      
            if prefix == target:        
                return True


        col_sums = [sum(grid[r][c] for r in range(m)) for c in range(n)]


        prefix = 0
        for j in range(n - 1):  
                   
            prefix += col_sums[j]      
            if prefix == target:
                return True


        return False
            
```

## Complexity

**Time Complexity:** $O(m \times n)$

We touch each cell a constant number of times: once for the total sum, once during the horizontal scan, and once when building `col_sums`. Everything else is $O(n)$ or $O(m)$ passes over pre-computed values.

**Space Complexity:** $O(n)$

We store the `col_sums` array of length $n$. No other data structures scale with input size.

## Edge Cases

**1. Odd total sum:** If the grid values sum to an odd number (like 7), you can immediately return `False`. No integer split can produce equal halves. This early exit saves unnecessary scanning.

**2. Single row or single column:** If `m == 1`, there are no valid horizontal cuts (you'd need at least 2 rows to have two non-empty sections). The loop `range(m - 1)` naturally handles this by iterating zero times. Same logic applies when `n == 1` for vertical cuts.

**3. Extremely large grids with constraints:** The problem says `2 <= m * n <= 10^5`, so one dimension can be 1 and the other up to 100,000. Your solution must not be $O(m \times n^2)$ or similar. Pre-computing `col_sums` protects you here.

**4. All elements equal:** Something like `[[2, 2], [2, 2]]` (total = 8, target = 4). The horizontal cut after row 0 gives prefix = 4 = target. Works perfectly, and is worth tracing manually to build confidence.


## Lessons

**Prefix sum is your best friend for "split at position k" problems.** Any time a problem asks "is there a cut point where the left sum equals the right sum", translate it to "is there a position where the prefix sum equals half the total". That reframe turns an $O(n^2)$ brute force into a clean $O(n)$ scan.

**Pre-compute aggregates across the axis you care about.** For vertical cuts, a column is not a contiguous memory chunk, so computing it on the fly inside the scan loop would be $O(m)$ per column, giving $O(m \times n)$ total work just for that loop. Building `col_sums` once upfront keeps you at $O(m \times n)$ overall without the hidden constant.

**Odd-sum early exit is a small but important habit.** Problems involving equal partition very frequently have this gate at the top. Training yourself to check parity first is a clean pattern that shows up in Partition Equal Subset Sum (416), Split Array with Equal Sum, and many others.
