# 1622. Fancy Sequence 🔴

**Topics:** Math, Design, Segment Tree


## Intuition

Imagine you're a bank manager. You have a list of customer account balances. Now your boss says:

> "Add $50 to **everyone's** account" or "Double **everyone's** balance"

You wouldn't go through every single account one by one, right? That would take forever if you have millions of customers. Instead, you'd just **write a note**: "everyone gets +50" or "everyone gets x2", and only apply it when someone actually asks for their specific balance.

That's exactly the trick here. Instead of touching every element on `addAll` or `multAll`, we **lazily track** a global multiplier and adder, and only resolve the true value when `getIndex` is called.


## Approach

The core idea is to maintain two global lazy values:
- `mult` (global multiplier, starts at 1)
- `add` (global adder, starts at 0)

At any point in time, the "true" value of an element is:

$$\text{truevalue} = \text{storedvalue} \times \text{mult} + \text{add}$$

**The catch with `append`:**

When we append a new value `val`, the global `mult` and `add` are already some values. We need to store a version of `val` that, when the formula is applied later, gives back `val`.

So we solve for `storedvalue`:

$$val = \text{storedvalue} \times \text{mult} + \text{add}$$

$$\text{storedvalue} = (val - \text{add}) \times \text{mult}^{-1} \mod (10^9 + 7)$$

This is where **modular inverse** comes in. Since $10^9 + 7$ is prime, by Fermat's Little Theorem:

$$a^{-1} \equiv a^{p-2} \pmod{p}$$



## Example Walkthrough

```
append(2)   -> mult=1, add=0  -> stored = (2-0) * inv(1) = 2
addAll(3)   -> add = 0*1 + 3 = 3
append(7)   -> stored = (7-3) * inv(1) = 4
multAll(2)  -> mult = 1*2 = 2, add = 3*2 = 6
getIndex(0) -> 2 * 2 + 6 = 10  ✓
```

## Code

```python
class Fancy:
    MOD = 10**9 + 7

    def __init__(self):
        self.seq = []
        self.mult = 1   
        self.add = 0     


    def mod_inv(self, a):
        # This works because MOD is prime
        return pow(a, self.MOD - 2, self.MOD)



    def append(self, val: int) -> None:
        normalized = (val - self.add) * self.mod_inv(self.mult) % self.MOD
        self.seq.append(normalized)




    def addAll(self, inc: int) -> None:
        # Instead of adding inc to every element, just shift the global adder
        self.add = (self.add + inc) % self.MOD




    def multAll(self, m: int) -> None:
        # Scale both the multiplier and adder globally
        self.mult = (self.mult * m) % self.MOD
        self.add = (self.add * m) % self.MOD



    def getIndex(self, idx: int) -> int:
        if idx >= len(self.seq):
            return -1
        # Apply the current global transform to recover the true value
        return (self.seq[idx] * self.mult + self.add) % self.MOD

        

# Your Fancy object will be instantiated and called as such:

# param_4 = obj.getIndex(idx)
```



## Complexity

| Operation | Time | Space |
|---|---|---|
| `append` | $O(\log p)$ for `pow` | $O(1)$ |
| `addAll` | $O(1)$ | $O(1)$ |
| `multAll` | $O(1)$ | $O(1)$ |
| `getIndex` | $O(1)$ | $O(1)$ |
| Overall space | | $O(n)$ |

The $O(\log p)$ in append comes from `pow(a, p-2, MOD)` (fast exponentiation).



## Edge Cases

1. **`getIndex` on out-of-bounds index:** If `idx >= len(seq)`, return `-1` immediately. Easy to miss if you assume the index is always valid.

2. **`multAll(0)` before an append:** If `mult` becomes 0, then `modinv(0)` is undefined! But the constraints say $m \geq 1$, so we're safe here. Worth noting though.

3. **Large number of operations:** With up to $10^5$ calls and values up to 100, intermediate values can balloon astronomically without `% MOD` at every step. Always reduce modulo at each operation, not just at the end.

4. **Single element sequence with multiple transforms:** Make sure `append` correctly captures the snapshot of `mult` and `add` at the time of insertion, not the latest values.



## Lessons

- **Lazy propagation is powerful.** Instead of updating $n$ elements on a global operation, track the transformation and apply it on demand. This pattern appears in segment trees too.

- **Modular inverse unlocks "undo" in modular arithmetic.** Whenever you need to reverse a multiplication under a modulus, reach for Fermat's Little Theorem: $a^{-1} \equiv a^{p-2} \pmod{p}$ (only valid when $p$ is prime).

- **Normalize on insert.** The key insight is storing values in a "canonical" form that is independent of the current global state. Then the global transform always applies cleanly on retrieval.

- **Think in terms of invariants.** Ask yourself: "What stays true no matter what operations are applied?" Here the invariant is: `truevalue = stored * mult + add`. Maintaining that invariant is the whole solution.
