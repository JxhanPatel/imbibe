# 874. Walking Robot Simulation 🟡 

**Topics:** Array, Hash Set, Simulation



## Intuition

Imagine you're playing with a toy RC car on a giant parking lot. The car starts facing north. You have a remote with three buttons: turn left, turn right, and go forward X steps. Some squares on the ground have big cones (obstacles) you can't drive through.

As you drive around, you keep mentally noting "what's the farthest I've ever been from the starting spot?" That's exactly what this problem asks. The twist is: you don't need the actual distance, just the squared version, which saves us a `sqrt()` call.


## Approach

We simulate the robot's movement step by step. The key insight is: **we check for obstacles one step at a time**, not all at once. So if a command says "move 5 steps", we move one step, check for a wall, move another, check again, and so on.

For obstacle lookup to be fast (instead of scanning the whole list every step), we throw all obstacles into a **hash set of tuples**. This makes each collision check $O(1)$ instead of $O(n)$.

The four directions (north, east, south, west) can be stored as $(dx, dy)$ pairs in clockwise order. Turning right means moving one index forward in that list, turning left means one index back. Clean and no messy angle math.



## Example Walkthrough

Input: `commands = [4, -1, 4, -2, 4]`, `obstacles = [[2, 4]]`

```
Start: pos = (0,0), facing North (index 0)
dirs = [(0,1), (1,0), (0,-1), (-1,0)]
         N        E       S        W

Command 4 (move north 4):
  step to (0,1) -> no obstacle, max = 1
  step to (0,2) -> no obstacle, max = 4
  step to (0,3) -> no obstacle, max = 9
  step to (0,4) -> no obstacle, max = 16

Command -1 (turn right):
  dir index: 0 -> 1, now facing East

Command 4 (move east 4):
  step to (1,4) -> no obstacle, max = 17
  step to (2,4) -> OBSTACLE! stay at (1,4)
  stop moving

Command -2 (turn left):
  dir index: 1 -> 0, now facing North

Command 4 (move north 4):
  step to (1,5) -> no obstacle, max = 26
  step to (1,6) -> no obstacle, max = 37
  step to (1,7) -> no obstacle, max = 50
  step to (1,8) -> no obstacle, max = 65

Answer: 65
```





## Code


```python
class Solution:
    def robotSim(self, commands: List[int], obstacles: List[List[int]]) -> int:

        # clockwise:- N, E, S, W
        dirs = [(0, 1), (1, 0), (0, -1), (-1, 0)]
        
        obs = set(map(tuple, obstacles))
        
        x, y = 0, 0

        dir_idx = 0  # starts facing north
        max_dist = 0
        

        for cmd in commands:
            if cmd == -1:
                dir_idx = (dir_idx + 1) % 4  # turn right

            elif cmd == -2:
                dir_idx = (dir_idx - 1) % 4  # turn left

            else:
                dx, dy = dirs[dir_idx]
                
                for _ in range(cmd):
                    nx, ny = x + dx, y + dy
                    
                    if (nx, ny) in obs:
                        break  # blocked, stop here

                    x, y = nx, ny
                    max_dist = max(max_dist, x*x + y*y)
        
        return max_dist
```




## Complexity

**Time:** $O(n + k \cdot m)$ where $n$ is the number of obstacles (for building the set), $k$ is the number of commands, and $m$ is the max steps per command (at most 9). So practically $O(n + k)$.

**Space:** $O(n)$ for the obstacle hash set.



## Edge Cases

**1. Obstacle at (0, 0):** The problem says the robot ignores it at the start but can't return. Our code naturally handles this since we only check `(nx, ny)` before stepping, never the starting position.

**2. No obstacles:** The set is empty, every lookup misses cleanly, and we just track the farthest point normally.

**3. Robot turns multiple times in a row (like `[-1, -1, -1]`):** No movement happens, just direction updates. The `else` branch never fires, so `x, y` stay unchanged. No issues there.

**4. Robot blocked on the first step of a command:** It stays put. `break` fires immediately after the first blocked check, and we move on to the next command without updating position.

**5. Negative coordinates:** The robot can drift into negative X/Y territory. Squaring handles this automatically since $(-3)^2 = 9$, same as $3^2$.



## Lessons

**1. Precompute for lookups:** Throwing obstacles into a set is the whole trick here. Without it, you'd be doing $O(n)$ checks per step and the solution would TLE.

**2. Direction cycling with modulo:** Storing directions as a list and using `(index + 1) % 4` for right turns and `(index - 1) % 4` for left turns is a classic pattern. Clean, no if/else spaghetti.

**3. Step-by-step simulation beats bulk math:** When obstacles are involved, you can't just compute "final position after 5 steps north" in one formula. You have to walk each step and check. Embrace the simulation.

**4. Update max during movement, not after:** You want the farthest point ever reached, which might be mid-command before getting blocked. Updating inside the inner loop captures that correctly.
