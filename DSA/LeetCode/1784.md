# 1784: Check if Binary String Has at Most One Segment of Ones 🟢
**Topics:** String  



## Intuition

Imagine you're looking at a row of houses on a street. Some houses have their lights on (represented by `1`), others are dark (`0`). The question is asking: do all the lit houses form **one continuous block**, or are there multiple separate groups of lit houses scattered around?

Think of it like this: if you walk down the street from left to right, you should see at most **one transition** from "lights on" to "lights off." Once the lights go off, they should stay off. If you see lights come back on after they've been off, that's a second segment. That's exactly what we're checking for in this binary string!



## Approach

We'll use a simple linear scan with a flag to track whether we've already seen the end of our first segment of ones.

Here's the insight: since the string **has no leading zeros** (guaranteed by constraints), any `1` we encounter at the start is part of our first segment. We keep reading `1`s until we hit a `0`. Once we hit that first `0`, we've "closed" our first segment. 

Now, if we ever see another `1` after that, it means there's a second segment of ones. We immediately return `false`. If we finish scanning without finding any `1` after the first `0`, we return `true`.

It's like reading a book: once you finish Chapter 1 (the ones) and see the chapter break (a zero), you shouldn't see Chapter 1 starting again!



## Example Walkthrough

Let's trace through `s = "1001"`:

| Index | Char | State | Explanation |
|-------|------|-------|-------------|
| 0 | `1` | In first segment | Starting our block of ones |
| 1 | `0` | First segment ended | We hit our first zero, flag this |
| 2 | `0` | Still after first segment | More zeros, no problem |
| 3 | `1` | **ALERT!** | Found a `1` after zeros. This is a second segment! |

Result: `false` (two separate segments: `1` at index 0 and `1` at index 3)

Now `s = "110"`:

| Index | Char | State |
|-------|------|-------|
| 0 | `1` | In first segment |
| 1 | `1` | Still in first segment |
| 2 | `0` | First segment ends |

We reach the end without seeing another `1`. Result: `true`



## Code

```python
class Solution:
    def checkOnesSegment(self, s: str) -> bool:

        seen_zero = False
        

        for char in s:

            if char == '0':
                seen_zero = True

            else: 

                if seen_zero:
                    return False
        
        return True
```


## Complexity

- **Time Complexity:** $O(n)$ where $n$ is the length of the string. We make a single pass through the characters.
- **Space Complexity:** $O(1)$. We only use one boolean variable regardless of input size.



## Edge Cases

| Case | Input | Expected | Why It Matters |
|------|-------|----------|----------------|
| **Single character** | `"1"` | `true` | Minimal valid input. No zeros at all means one segment. |
| **All ones** | `"11111"` | `true` | Never see a zero, so never exit the first segment. |
| **One then all zeros** | `"1000"` | `true` | First segment ends early, but no second segment starts. |
| **Alternating** | `"101010"` | `false` | Multiple segments of single `1`s separated by `0`s. |
| **Two segments** | `"111000111"` | `false` | Classic case: clear gap between two groups of ones. |



## Lessons

1. **State Machine Thinking:** This problem is really about tracking state (are we in the first segment? have we exited it?). Many string problems become clearer when you think in terms of state transitions.

2. **Leverage Constraints:** The "no leading zeros" constraint is crucial. It guarantees we start inside the first segment, so we don't need to "find" it first. Always read constraints carefully. They often simplify your logic dramatically.

3. **Early Exit:** Notice how we return `False` immediately when we detect a violation. We don't need to scan the rest of the string. This is a common optimization pattern.

4. **The "Once X happens, Y should never happen" Pattern:** This comes up frequently (e.g., "once sorted, stay sorted", "once balanced, don't go negative"). A simple flag variable is often the cleanest solution.
