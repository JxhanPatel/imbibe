# 3661: Maximum Walls Destroyed by Robots 🔴

**Topics:** Sorting, Binary Search, Prefix Sums



## Intuition

Imagine you're at a bowling alley, but the lanes have **bumpers (robots)** scattered around. Each player (robot) gets one ball and can throw it left or right. The ball rolls until it either hits another player or hits the end of its range. Pins (walls) that the ball passes through get knocked down.

The key insight: a robot's bullet flies in one direction and stops the moment it hits another robot (not a wall). So for each robot, you need to figure out:
- How far left can its bullet actually go before hitting another robot?
- How far right can it go before hitting another robot?

Then, given those real left/right limits, count how many walls fall in between. You want to pick the direction (left or right) per robot that knocks down the most walls total.

But wait, each wall is only counted once no matter how many robots could hit it. That makes this a **"maximize unique coverage"** problem, which is the actual hard part.



## Approach


1. **Sort robots by position.** Once sorted, each robot's nearest left neighbor and nearest right neighbor are just the adjacent robots in the array. That makes finding the bullet's real travel range easy.

2. **For each robot, compute its left and right firing ranges:**
   - Left range: `[robot_pos - distance, robot_pos]` clipped to stop at the previous robot's position.
   - Right range: `[robot_pos, robot_pos + distance]` clipped to stop at the next robot's position.

3. **Sort walls by position** and build a **prefix sum array** so that for any range `[L, R]`, you can count walls inside it in O(log n) time using binary search.

4. **For each robot, pick the direction** (left or right) that hits more walls. Add those walls to a set (or use smart counting to avoid double-counting).

5. Wait, step 4 still has the double-counting problem. The trick: **you can't naively sum per-robot maximums** because two robots might both "claim" the same wall. So instead, think of it differently:

   Each robot's left coverage and right coverage are **non-overlapping intervals** across all robots (since the intervals are bounded by adjacent robots). This means walls in different robots' ranges never overlap between robots. So you can safely sum.

   But within a single robot, you pick the max of left vs right. Since left and right don't overlap (they share only the robot's position), the union is just left_count + right_count and you pick the better side.

6. Return the total sum of chosen (left or right) wall counts across all robots.



## Example Walkthrough

Input: `robots = [10, 2]`, `distance = [5, 1]`, `walls = [5, 2, 7]`

**Step 1: Sort robots by position**
- `sorted_robots = [(2, dist=1), (10, dist=5)]`

**Step 2: Sort walls**
- `sorted_walls = [2, 5, 7]`
- prefix counts: at index 0 is wall 2, index 1 is wall 5, index 2 is wall 7

**Step 3: Ranges for robot at pos=2, dist=1**
- Left range: `[2-1, 2]` = `[1, 2]`. Left neighbor? None (it's the first). So range stays `[1, 2]`.
- Right range: `[2, 2+1]` = `[2, 3]`. Right neighbor is at pos=10. Range stays `[2, 3]`.
- Walls in `[1, 2]`: wall 2 → count = 1
- Walls in `[2, 3]`: wall 2 → count = 1
- Pick max = 1 (either direction)

**Step 4: Ranges for robot at pos=10, dist=5**
- Left range: `[10-5, 10]` = `[5, 10]`. Left neighbor is at pos=2. Range starts after 2, so stays `[5, 10]`.
- Right range: `[10, 10+5]` = `[10, 15]`. Right neighbor? None. Range stays `[10, 15]`.
- Walls in `[5, 10]`: walls 5 and 7 → count = 2
- Walls in `[10, 15]`: none → count = 0
- Pick max = 2 (fire left)

**Total: 1 + 2 = 3** ✓



## Code

```python
class Solution:
    def maxWalls(self, robots: List[int], distance: List[int], walls: List[int]) -> int:

        def lower_bound(arr, target):
            lo, hi = 0, len(arr)
            while lo < hi:
                mid = (lo + hi) // 2
                if arr[mid] < target:
                    lo = mid + 1
                else:
                    hi = mid
            return lo

        def upper_bound(arr, target):
            lo, hi = 0, len(arr)
            while lo < hi:
                mid = (lo + hi) // 2
                if arr[mid] <= target:
                    lo = mid + 1
                else:
                    hi = mid
            return lo

        # Map robot -> distance (kept simple; inputs assumed consistent)
        dist_map = {robots[i]: distance[i] for i in range(len(robots))}

        robots.sort()
        walls.sort()

        n = len(robots)

        left = [0] * n
        right = [0] * n
        between = [0] * n  # walls between consecutive robots

        for i in range(n):
            r = robots[i]
            d = dist_map[r]

            # left side window
            left_limit = r - d
            if i > 0:
                left_limit = max(left_limit, robots[i - 1] + 1)

            l_idx = lower_bound(walls, left_limit)
            r_idx = upper_bound(walls, r)
            left[i] = r_idx - l_idx

            # right side window
            right_limit = r + d
            if i < n - 1:
                right_limit = min(right_limit, robots[i + 1] - 1)

            r_bound = upper_bound(walls, right_limit)
            l_bound = lower_bound(walls, r)
            right[i] = r_bound - l_bound

            # overlap with previous robot's segment
            if i > 0:
                prev_idx = lower_bound(walls, robots[i - 1])
                between[i] = r_idx - prev_idx

        # DP accumulation
        best_left = left[0]
        best_right = right[0]

        for i in range(1, n):
            take_left = best_left + left[i]

            merge = best_right - right[i - 1]
            merge += min(left[i] + right[i - 1], between[i])

            new_left = max(take_left, merge)
            new_right = max(best_left + right[i], best_right + right[i])

            best_left, best_right = new_left, new_right

        return max(best_left, best_right)
```



## Complexity

**Time:** Sorting robots and walls costs $O(n \log n + m \log m)$ where $n$ = number of robots and $m$ = number of walls. For each robot, we do two binary searches so the main loop is $O(n \log m)$. Overall: $O((n + m) \log (n + m))$.

**Space:** $O(n + m)$ for sorting (Python's Timsort is in-place on lists, but `sorted()` creates new lists).



## Edge Cases

- **Robot at the very edge (no left or right neighbor):** The range is unbounded on that side, limited only by `distance`. The code handles this with the `if i > 0` and `if i < len(bots) - 1` guards.

- **Robot blocked on both sides (neighbor closer than range):** e.g., two robots at positions 2 and 3 each with distance 100. Robot at 2 can only fire right to position 2 (stopped by robot at 3) and left freely. Binary search still gives the right answer since the range simply collapses.

- **Wall at the same position as a robot:** The problem says this is valid; the bullet can hit the wall even from the robot's own position. `count_walls(lo, pos)` includes `pos` in the range, so it works.

- **Multiple walls in the same range (both left and right could claim them):** Within a single robot, left range ends at `pos` and right range starts at `pos`. They share at most `pos` itself, but the binary search counts it in exactly one direction. Across different robots, ranges are bounded by robot positions so they never truly overlap, meaning no double-counting.

- **All robots in a line with distance 1:** Most will be completely blocked and can only hit walls exactly at their own position. This tests that the clipping math is correct when `pos - dist` and `pos + dist` both fall inside neighboring robots.



## Lessons

- **Sorting unlocks neighbor lookups in O(1).** Once robots are sorted, adjacent robots in the sorted list are the only ones that can block a bullet. No need for complex spatial queries.

- **Prefix sums / binary search for range counts.** Anytime you need "how many values fall in [L, R]" repeatedly, sort the values and use `bisect_left` / `bisect_right`. It's one of those patterns you'll use over and over.

- **Non-overlapping intervals let you safely sum.** The reason this problem avoids the double-counting trap is the structure of the intervals (bounded by neighbors). Recognizing when intervals are guaranteed non-overlapping is a key insight that unlocks a clean summation.

- **Left boundary vs right boundary asymmetry.** The left range includes the robot's own position; be precise about whether bounds are open or closed. Off-by-one errors with `bisect_left` vs `bisect_right` are the most common bug here.
