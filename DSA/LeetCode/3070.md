# 3070. Count Submatrices with Top-Left Element and Sum Less Than k 🟡 
**Topics:** Array, Matrix, Prefix Sum



## Intuition

Imagine you are a city planner looking at a map of districts on a grid. You always start from the top-left corner of the city, and you want to count how many rectangular sections of the city (always anchored to that top-left corner) have a total "development cost" at or under your budget `k`.

The naive way is to redraw every rectangle from scratch and tally costs one cell at a time. That is slow. The smart way is to precompute a "running total" map where each cell already stores the total cost of the entire rectangle from the top-left down to that cell. Then you can check any rectangle's cost in O(1) with a simple lookup.

That running total map is called a **2D prefix sum**.



## Approach

We build a prefix sum table `ps` in-place (or on a copy of the grid) where `ps[i][j]` stores the sum of all elements in the rectangle from `(0,0)` to `(i,j)`. The formula for building it is:

```
ps[i][j] = grid[i][j] + ps[i-1][j] + ps[i][j-1] - ps[i-1][j-1]
```

Why subtract `ps[i-1][j-1]`? Because when you add the row-above and the column-left prefix sums, you count the top-left rectangle (the overlap) twice. Subtracting once fixes that.

Once built, every cell `ps[i][j]` already represents the sum of a submatrix anchored at `(0,0)`. So we simply walk every cell and check if `ps[i][j] <= k`. If yes, count it.



## Example Walkthrough

```
grid = [[7,6,3],[6,6,1]], k = 18
```

**Step 1: Build prefix sum table**

Start at `(0,0)`: ps = 7
Move right to `(0,1)`: 7 + 6 = 13
Move right to `(0,2)`: 13 + 3 = 16

Move down to `(1,0)`: 7 + 6 = 13
Move to `(1,1)`: 6 + 13 + 13 - 7 = 25
Move to `(1,2)`: 1 + 16 + 25 - 13 = 29

```
Prefix sum grid:
[[ 7, 13, 16],
 [13, 25, 29]]
```

**Step 2: Count cells where ps[i][j] <= 18**

- ps[0][0] = 7   ✅
- ps[0][1] = 13  ✅
- ps[0][2] = 16  ✅
- ps[1][0] = 13  ✅
- ps[1][1] = 25  ❌
- ps[1][2] = 29  ❌

**Answer: 4**



## Code

```python
class Solution:
    def countSubmatrices(self, grid: List[List[int]], k: int) -> int:

        m = len(grid)
        n = len(grid[0])

        count = 0

        # builds the 2D prefix sum directly in grid to save space
        for i in range(m):
            for j in range(n):


                # add the cell above if it exists
                if i > 0:
                    grid[i][j] += grid[i-1][j]


                # add the cell to the left if it exists
                if j > 0:
                    grid[i][j] += grid[i][j-1]


                # we added the top-left overlap region twice, so subtract it once
                if i > 0 and j > 0:
                    grid[i][j] -= grid[i-1][j-1]


                # every cell now = sum of top-left submatrix ending here
                # if it's within budget, count it
                if grid[i][j] <= k:
                    count += 1

        return count
```

> Note: This modifies the input grid. If that is a concern in your environment, make a deep copy first with `grid = [row[:] for row in grid]`.



## Complexity

**Time:** $O(m \times n)$ - we visit every cell exactly once.

**Space:** $O(1)$ - we reuse the grid itself. If we used a separate `ps` table it would be $O(m \times n)$ extra space.



## Edge Cases

**1. Single cell grid:** `grid = [[5]], k = 10` - the loop runs once, `ps[0][0] = 5 <= 10`, returns 1. Works correctly since the boundary checks (`if i > 0`, `if j > 0`) guard against out-of-bounds access.

**2. Top-left cell already exceeds k:** `grid = [[100, 1]], k = 5` - `ps[0][0] = 100 > 5`, so we count 0 from the start. Every other submatrix must include `(0,0)`, so their sums can only grow. The answer is 0.

**3. All values are zero:** `grid = [[0,0],[0,0]], k = 0` - every prefix sum stays 0, so all 4 submatrices are counted. Make sure your condition is `<= k`, not `< k`.

**4. Very large grid at max constraints (1000x1000):** Values up to 1000 per cell, so the maximum possible prefix sum is $1000 \times 1000 \times 1000 = 10^9$, which fits comfortably in a 32-bit int (max ~$2.1 \times 10^9$) and well within Python's arbitrary precision integers.



## Lessons

**1. 2D prefix sums are your best friend for rectangle sum queries.** Any time a problem asks "sum of a subgrid", reach for this pattern first. The recurrence `ps[i][j] = val + top + left - top-left` is worth memorizing cold.

**2. Inclusion-exclusion is the reason the formula works.** The subtraction of the diagonal overlap is a classic counting correction. You will see this same idea in problems involving set unions too.

**3. The constraint "must include (0,0)" is what makes this problem approachable.** For arbitrary submatrices, you would need a more complex approach. The top-left anchor means every valid submatrix is exactly the prefix rectangle ending at some `(i,j)`, so checking all `m*n` cells is all you need.

**4. In-place modification is a space trick, not always appropriate.** If the problem guaranteed read-only input, you would allocate a separate table. Know when to apply it and when to skip it.
