# 1665. Minimum Initial Energy to Finish Tasks 🔴

**Topics:** Greedy, Sorting, Array






## Intuition

Imagine each task is like entering a game level.

* `minimum[i]` = the minimum stamina required to even enter the level
* `actual[i]` = the stamina actually consumed after completing it

Some levels are "strict" because they require a lot of energy upfront compared to what they consume.

So the key idea is:

> Finish the most demanding tasks first while your energy is high.

A task with a large gap:

$$
minimum - actual
$$

is dangerous to postpone because later your energy may become too low to start it.

So we greedily perform tasks with larger gaps first.







## Approach

### Step 1: Sort Tasks

Sort tasks by:

$$
(minimum - actual)
$$

in descending order.

Why?

Because tasks that require much more starting energy than they consume should be completed earlier.


### Step 2: Simulate the Process

Maintain:

* `energy` → current energy
* `answer` → extra energy added initially

For every task:

* If current energy is less than required minimum:

  increase both `answer` and `energy`.

* Then complete the task and subtract actual cost.







## Example Walkthrough

Input:

```python
tasks = [[1,2],[2,4],[4,8]]
```

### Sort by `(minimum - actual)`

| Task  | Gap |
| ----- | --- |
| [4,8] | 4   |
| [2,4] | 2   |
| [1,2] | 1   |

Sorted order:

```python
[[4,8],[2,4],[1,2]]
```



Start:

```python
energy = 0
answer = 0
```

### Task [4,8]

Need 8 energy, have 0.

Add 8 energy.

```python
answer = 8
energy = 8
```

Complete task:

```python
energy = 8 - 4 = 4
```



### Task [2,4]

Already have 4 energy.

Complete task:

```python
energy = 4 - 2 = 2
```



### Task [1,2]

Already have 2 energy.

Complete task:

```python
energy = 2 - 1 = 1
```

Finished successfully.

Final answer:

```python
8
```









## Code

```python
from typing import List

class Solution:
    def minimumEffort(self, tasks: List[List[int]]) -> int:

        tasks.sort(key=lambda t: t[1] - t[0], reverse=True)
        
        curr=0
        total=0

        for cost, needed in tasks:
            
            if curr < needed:
                total += needed-curr
                curr=needed

            curr-= cost
        
        return total
```







## Complexity

### Time Complexity

Sorting dominates:

$$
O(n \log n)
$$

### Space Complexity

$$
O(1)
$$

Ignoring sorting space.












## Edge Cases

### 1. Only One Task

```python
tasks = [[5,10]]
```

Need exactly `10`.


### 2. Tasks Where `actual == minimum`

```python
tasks = [[3,3],[4,4]]
```

Order does not matter much.


### 3. Very Large Constraints

Up to:

$$
10^5
$$

tasks.

An (O(n^2)) solution would timeout, so sorting + greedy is necessary.







## Lessons

### 1. Greedy Ordering Matters

Sometimes choosing the correct order reduces the required starting resource dramatically.



### 2. Focus on the "Penalty"

The important value is:

$$
minimum - actual
$$

This tells us how restrictive a task is.



### 3. Simulating Backwards Thinking

Instead of asking:

> "Can I finish from this energy?"

we ask:

> "How much extra energy must I add to make this possible?"

That small perspective shift makes the greedy simulation simple.
