# 3600. Maximize Spanning Tree Stability with Upgrades 🔴

**Topics:** Graph, Greedy, Binary Search, Union-Find, Spanning Tree



## Intuition

Imagine you are building a road network connecting several cities. Some roads are **mandatory** (government contracts, cannot be changed), while others are **optional** and you can pay to "upgrade" them (widen the road, increasing its capacity). You have a limited budget of `k` upgrades.

The "stability" of your network is only as strong as its **weakest road**. So your goal is to pick roads such that even the worst road in your network is as strong as possible.

This is a classic **"maximize the minimum"** pattern, which almost always screams **binary search on the answer**. You binary search on what the minimum strength could be, and check: "Can I build a valid spanning tree where every edge has at least this strength?"



## Approach

Here is the plan:

**Step 1: Binary Search on the answer**
The answer (stability) must be one of the possible edge strengths, either original or doubled. So we collect all candidate values and binary search over them.

**Step 2: For each candidate minimum strength `mid`, check feasibility**
Given a target stability `mid`, we want to know if we can form a spanning tree where every included edge has strength >= `mid`.

The check works greedily using **Union-Find (DSU)**:
1. First, include all **mandatory edges**. If a mandatory edge has strength < `mid`, it fails immediately (we cannot upgrade mandatory edges).
2. Among optional edges, an edge with strength `s` is "good" if `s >= mid` (no upgrade needed) or `2*s >= mid` (needs one upgrade). An edge with `2*s < mid` is useless no matter what.
3. Greedily add "good without upgrade" edges first, then "good with upgrade" edges (using up our `k` budget), to try to connect all nodes.

**Why greedy works here:** We want to minimize upgrades used while connecting the graph. Preferring no-upgrade edges first conserves upgrades for edges that actually need them.



## Example Walkthrough

**Input:** `n=3, edges=[[0,1,4,0],[1,2,3,0],[0,2,1,0]], k=2`

Candidate strengths: `{1, 2, 3, 4, 6, 8}` (original + doubled values)

Binary search lands on checking `mid = 6`:
- No mandatory edges
- Edge [0,1,4]: needs upgrade (4 < 6, but 8 >= 6) -- upgrade candidate
- Edge [1,2,3]: needs upgrade (3 < 6, but 6 >= 6) -- upgrade candidate  
- Edge [0,2,1]: useless (1 < 6, 2 < 6) -- skip

Add "free" edges first: none qualify.
Add upgrade edges: use [0,1] (upgrade 1), use [1,2] (upgrade 2). Now all 3 nodes connected!
Upgrades used = 2 <= k = 2. Feasible! Try higher.

Check `mid = 8`:
- Edge [0,1,4]: upgrade gives 8 >= 8, ok
- Edge [1,2,3]: upgrade gives 6 < 8, useless
- Edge [0,2,1]: useless

Can only connect 0-1, cannot reach node 2. Not feasible.

**Answer = 6**



## Code

```python
class Solution:
    def maxStability(self, n: int, edges: List[List[int]], k: int) -> int:

        def check(min_s):
            parent = list(range(n))


            def find(x):
                while parent[x] != x:
                    parent[x] = parent[parent[x]]
                    x = parent[x]

                return x


            def union(x, y):
                px, py = find(x), find(y)
                if px == py: return False
                parent[px] = py

                return True



            components, upgrades = n, 0

            # pass 1: mandatory edges
            for u, v, s, m in edges:
                if m == 1:
                    if s < min_s or not union(u, v): return False
                    components -= 1



            # pass 2: free optional edges (no upgrade needed)
            for u, v, s, m in edges:
                if m == 0 and s >= min_s and union(u, v):
                    components -= 1



            # pass 3: upgrade optional edges (costs from budget)
            for u, v, s, m in edges:
                if m == 0 and s < min_s <= s*2 and upgrades < k and union(u, v):
                    components -= 1
                    upgrades += 1

            return components == 1



        if not check(0): return -1

        candidates = sorted({v for u,v,s,m in edges for v in (s, s*2)})
        lo, hi, res = 0, len(candidates)-1, candidates[0]


        while lo <= hi:
            mid = (lo + hi) // 2
            if check(candidates[mid]): res = candidates[mid]; lo = mid+1
            else: hi = mid-1

        return res
```



## Complexity

**Let** $E$ = number of edges, $N$ = number of nodes.

- **Collecting candidates:** $O(E)$
- **Binary search:** $O(\log E)$ iterations
- **Each feasibility check:** $O(E \cdot \alpha(N))$ where $\alpha$ is the inverse Ackermann function (nearly constant) from DSU operations

**Total Time:** $O(E \cdot \alpha(N) \cdot \log E)$

**Space:** $O(N + E)$ for DSU and edge lists


## Edge Cases

| Scenario | What happens |
|---|---|
| Mandatory edges form a cycle | `union` returns False for a mandatory edge, immediately return -1 |
| `k = 0` with weak optional edges | Upgrade edges are skipped entirely, only free edges and mandatory edges used |
| Graph is disconnected even with all edges | `can_achieve(0)` returns False, output is -1 |
| All edges are mandatory | No optional edges to upgrade; stability is the min strength among mandatory edges if they form a valid spanning tree |
| `k >= n-1` | Potentially all spanning tree edges can be upgraded, giving maximum flexibility |



## Lessons

**1. "Maximize the minimum" almost always means binary search on the answer.** Whenever you see this pattern, ask yourself: "Can I check feasibility for a given target value?" If yes, binary search on the candidates.

**2. Binary search on a discrete set of candidates, not a range.** The answer must be an actual edge strength (original or doubled). Searching over those values keeps it clean and correct.

**3. Greedy + DSU is the classic spanning tree toolkit.** When building spanning trees with constraints, always think about ordering edges by priority and using Union-Find to detect cycles efficiently.

**4. Separate "free" from "costly" actions.** Doing the free actions first before spending your budget is a greedy principle that works whenever spending budget and not spending are equivalent in terms of graph connectivity.

**5. Validate impossible cases early.** Checking `can_achieve(0)` at the start handles the disconnected graph case cleanly before the binary search even begins.
