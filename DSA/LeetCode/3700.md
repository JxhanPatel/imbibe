# 3700. Number of ZigZag Arrays II 🔴

**Topics:** Dynamic Programming, Matrix Exponentiation, State Compression


## Intuition

Imagine you're driving on a winding mountain road.

* If you just turned **uphill**, the next turn must go **downhill**.
* If you just turned **downhill**, the next turn must go **uphill**.
* Going straight at the same level twice in a row is not allowed.

A ZigZag array follows the same idea.

For any three consecutive numbers:

* They cannot keep increasing.
* They cannot keep decreasing.
* Adjacent values must be different.

The important observation is that when building the array, we do not need to remember the entire sequence.

We only need to know:

1. The current value.
2. Whether the last move was **up** or **down**.

Since the value range contains at most:

$$
r-l+1 \le 75
$$

the number of possible states is small.

However, `n` can be as large as:

$$
10^9
$$

which makes ordinary DP impossible.

This is a classic situation where **matrix exponentiation** helps us apply the same DP transition billions of times in logarithmic time.


## Approach

Let:

$$
m = r-l+1
$$

represent the number of distinct values.

### State Representation

For every value:

$$
0,1,\dots,m-1
$$

we maintain two states:

* `up[i]` = arrays ending at value `i` whose last move was upward.
* `down[i]` = arrays ending at value `i` whose last move was downward.

Total states:

$$
2m
$$


### Transition Matrix

We build a matrix `T` of size:

$$
(2m)\times(2m)
$$

that describes how states evolve.

#### Down → Up

If the previous move was downward and the new value is larger:

```text
prev < cur
```

then:

```text
down(prev) -> up(cur)
```

This is represented by:

```python
for cur in range(prev + 1, m):
    T[cur][m + prev] = 1
```


#### Up → Down

If the previous move was upward and the new value is smaller:

```text
cur < prev
```

then:

```text
up(prev) -> down(cur)
```

This is represented by:

```python
for cur in range(prev):
    T[m + cur][prev] = 1
```


### Initial State

We first construct all valid arrays of length `2`.

For every pair `(a,b)`:

* If `a < b`, it contributes to an `up` state.
* If `a > b`, it contributes to a `down` state.
* Equal values are ignored because adjacent elements cannot be equal.


### Matrix Exponentiation

The state vector already represents arrays of length `2`.

To reach length `n`, we need:

$$
n-2
$$

additional transitions.

Instead of applying them one by one, we compute:

$$
T^{n-2}
$$

using binary exponentiation.

This reduces the complexity from:

$$
O(n)
$$

to:

$$
O(\log n)
$$

matrix multiplications.


## Example Walkthrough

### Input

```text
n = 3
l = 1
r = 3
```

Values available:

```text
1, 2, 3
```


### Build Length-2 States

Valid increasing pairs:

```text
(1,2)
(1,3)
(2,3)
```

These contribute to `up` states.

Valid decreasing pairs:

```text
(2,1)
(3,1)
(3,2)
```

These contribute to `down` states.


### One More Transition

Consider:

```text
1 -> 3
```

The last move is upward.

The next value must be smaller than `3`:

```text
1 -> 3 -> 1
1 -> 3 -> 2
```

Applying this rule to every state produces all valid arrays of length `3`.

Final answer:

```text
10
```


## Code

```python
class Solution:

    MOD = 1_000_000_007

    def zigZagArrays(self, n: int, l: int, r: int) -> int:
        m = r-l+1

        if n==1:
            return m

        size = 2*m

        T = [[0] * size for _ in range(size)]


        for prev in range(m):
            for cur in range(prev+1, m):
                T[cur][m + prev] = 1

            for cur in range(prev):
                T[m + cur][prev] = 1


        state = [0]*size

        for a in range(m):
            for b in range(m):
                if a < b:
                    state[b] += 1

                elif a > b:
                    state[m + b] += 1

        def mat_mul(A, B):
            sz = len(A)
            res = [[0] * sz for _ in range(sz)]
            
            for i in range(sz):
                for k in range(sz):
                    if not A[i][k]:
                        continue

                    aik = A[i][k]
                    row = res[i]
                    brow = B[k]

                    for j in range(sz):
                        if brow[j]:
                            row[j] = (row[j] + aik * brow[j]) % self.MOD
            return res


        def mat_vec_mul(A, v):
            sz = len(v)
            res = [0]*sz

            for i in range(sz):
                t=0
                row = A[i]

                for j in range(sz):
                    if row[j] and v[j]:
                        t += row[j]*v[j]
                res[i] = t % self.MOD

            return res

        steps = n-2
        
        while steps:
            if steps & 1:
                state = mat_vec_mul(T, state)
            T = mat_mul(T, T)
            steps >>= 1

        return sum(state) % self.MOD
```


## Complexity

Let

$$
m = r-l+1
$$

and

$$
S = 2m
$$

be the number of states.

### Time Complexity

Matrix multiplication:

$$
O(S^3)
$$

Performed for:

$$
O(\log n)
$$

powers.

Overall:

$$
O(S^3 \log n)
$$

Since:

$$
S \le 150
$$

this is efficient enough.


### Space Complexity

Transition matrix:

$$
O(S^2)
$$

So the total space complexity is:

$$
O(S^2)
$$


## Edge Cases

### 1. Only Two Values

```text
n = 3
l = 4
r = 5
```

Possible arrays:

```text
[4,5,4]
[5,4,5]
```

Answer:

```text
2
```


### 2. Very Large n

```text
n = 10^9
```

A normal DP would require billions of iterations.

Matrix exponentiation handles it in:

$$
O(\log n)
$$

steps.


### 3. Adjacent Values Cannot Match

```text
[2,2,1]
```

Invalid because:

```text
2 == 2
```


### 4. Three Increasing Values

```text
[1,2,3]
```

Invalid because the sequence is strictly increasing.


### 5. Three Decreasing Values

```text
[5,4,2]
```

Invalid because the sequence is strictly decreasing.


## Lessons

* When `n` is extremely large, look for a way to model DP transitions as a matrix.
* State compression is often enough to reduce a seemingly impossible problem into a manageable number of states.
* Matrix exponentiation is essentially applying the same transition repeatedly in logarithmic time.
* Tracking only the **last direction** is sufficient because the ZigZag constraint depends only on the most recent comparison.
* Problems with a small state space and huge number of steps are strong indicators for matrix exponentiation.
