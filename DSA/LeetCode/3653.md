# 3653. XOR After Range Multiplication Queries I 🟡
**Topics:** Array, Simulation, Bitwise Operations




## Intuition
Imagine you have a row of lockers, each containing a certain number of coins. A supervisor comes in with a list of instructions. Each instruction tells them to start at a specific locker, skip a certain number of lockers (the step size), and multiply the coins in those specific lockers by a certain multiplier. 

Since the rules of this problem specify a modulo for the multiplication but want the **XOR** of the final totals, we can't use many shortcut "math tricks" for the XOR part until the multiplication is finished. Because the constraints are relatively small ($N$ and $Q$ both up to 1,000), we can follow the supervisor’s instructions literally.

## Approach
For this specific version of the problem, the most reliable way to ensure we handle the modulo math correctly is to perform a **Direct Simulation**.

1.  **Iterate through Queries:** We process each query one by one.
2.  **Step-based Loop:** For each query, we start at index $l_i$ and move to the right. Instead of checking every index, we increment our index by $k_i$.
3.  **Apply Multiplier:** We update the value at the current index by multiplying it by $v_i$. Crucially, we apply the modulo $10^9 + 7$ immediately to keep the numbers from growing into monsters that slow down the computer.
4.  **Calculate XOR:** Once all queries are done, we loop through the final state of the `nums` array and XOR all the values together.



## Example Walkthrough
**Input:** `nums = [1, 1, 1]`, `query = [[0, 2, 1, 4]]`

* **Initial Array:** `[1, 1, 1]`
* **Process Query [0, 2, 1, 4]:**
    * Start at index `0`. `nums[0] = (1 * 4) = 4`.
    * Move by step `1` to index `1`. `nums[1] = (1 * 4) = 4`.
    * Move by step `1` to index `2`. `nums[2] = (1 * 4) = 4`.
* **Final Array:** `[4, 4, 4]`
* **XOR Result:** $4 \oplus 4 \oplus 4 = 4$.





### Code

```python
class Solution:
    def xorAfterQueries(self, nums: List[int], queries: List[List[int]]) -> int:
        MOD = 1000000007
        

        for li, ri, ki, vi in queries:
            idx = li


            while idx <= ri:

                nums[idx] = (nums[idx] * vi) % MOD
                idx += ki
                

        ans = 0
        for x in nums:
            ans ^= x
            
        return ans
```



## Complexity

* **Time Complexity:** $O(Q \cdot \frac{N}{K} + N)$. In the worst case where $K=1$, this simplifies to $O(Q \cdot N)$. Given $N, Q \leq 1000$, $10^6$ operations is well within the limits for a typical 1-second execution time.
* **Space Complexity:** $O(1)$ if we don't count the input array, as we modify the array in place and only use a few extra variables.




## Edge Cases
1.  **Step size $k_i > n$:** If the step size is larger than the array length, the loop will only execute once for the starting index $l_i$. Our `while` loop handles this naturally.
2.  **Maximum $v_i$ and $nums[i]$:** The values can reach up to $10^5$ and $10^9$ respectively. Multiplying them can exceed the 32-bit integer limit, but Python handles large integers automatically. The modulo ensures we don't stay in "large" territory for long.
3.  **$l_i = r_i$:** The query only targets a single element. The loop runs once and then terminates.



## Lessons
* **Don't Over-engineer:** When constraints are small ($10^3$), a simple simulation is often better than a complex segment tree or lazy propagation logic. It's faster to write and easier to debug.
* **Modulo Placement:** Always apply the modulo *immediately* after a multiplication. If you wait until the end of the query, the numbers might become so large that they consume massive amounts of memory and CPU time.
* **XOR Property:** Remember that $x \oplus x = 0$ and $x \oplus 0 = x$. While not strictly needed for the simulation, it's the core reason why the final XOR result changes so drastically based on how many times an element was updated.
