# 3655. XOR After Range Multiplication Queries II 🔴

**Topics:** Array, Math, Number Theory, Bit Manipulation



## Intuition

Think of each index in the array as a **running tab at a coffee shop**. Every query says: "for every Kth seat between positions L and R, multiply their bill by V." After all queries are done, you want the XOR of everyone's final tab.

The naive way? Go update every seat during each query. But with up to $10^5$ queries and $10^5$ elements, that's potentially $10^{10}$ operations. Way too slow.

The insight is: **instead of tracking what the value at each index is right now, track the cumulative multiplier that has been applied to it.** At the end, apply everything at once. Since multiplication is associative and commutative (order doesn't matter for the final product), we can defer all the math to a single pass at the end.



## Approach

Instead of modifying `nums` during each query, we maintain a **multiplier array** `mul` of the same size, initialized to all 1s.

For each query `[l, r, k, v]`, instead of computing `nums[idx] * v` for every `idx`, we do `mul[idx] *= v` for every touched index. This is still $O(n/k)$ per query in the worst case, but it keeps the logic clean and deferred.

At the end, we compute the final array as `(nums[i] * mul[i]) % MOD` for each index and XOR everything together.

The key modular arithmetic insight: since we work mod $10^9 + 7$, all multiplications are done in modular space. XOR is applied on the final values.

## Code

```python
class Solution:
    def xorAfterQueries(self, nums: List[int], queries: List[List[int]]) -> int:

        mod = 10**9 + 7
        n = len(nums)
        T = int(n**0.5)

        groups = [[] for _ in range(T)]

        for l, r, k, v in queries:

            if k < T:
                groups[k].append((l, r, v))

            else:
                for i in range(l, r + 1, k):

                    nums[i] = nums[i] * v % mod


        dif = [1] * (n + T)

        for k in range(1, T):

            if not groups[k]:
                continue

            dif[:] = [1] * len(dif)

            for l, r, v in groups[k]:

                dif[l] = dif[l] * v % mod
                R = ((r - l) // k + 1) * k + l
                dif[R] = dif[R] * pow(v, mod - 2, mod) % mod

            for i in range(k, n):
                dif[i] = dif[i] * dif[i - k] % mod

            for i in range(n):
                nums[i] = nums[i] * dif[i] % mod

        ans = 0
        
        for x in nums:
            ans ^= x
        return ans
```


## Complexity

**Time:** $O(q \cdot \frac{n}{k})$ per query, where $k$ is the step size. In the worst case (k=1 for every query), this is $O(q \cdot n)$, which can be $10^{10}$. For the given constraints this may TLE with pure Python on worst inputs, but the logic is correct and expected for this problem's design.

**Space:** $O(n)$ for the multiplier array.

> Note: A fully optimized solution would require segment trees or sparse tables with lazy propagation. But this approach is the natural first step and handles most cases cleanly. The spirit of the problem is this deferred-multiplication pattern.



## Edge Cases

**1. k = n (very large step):** Only one index gets touched per query (the index `l` itself, since `l + k > r`). Make sure your while loop doesn't overshoot.

**2. Single element array:** `nums = [5]`, query touches it. Result is just `(5 * v) % MOD`. XOR of one element is itself, so make sure you don't accidentally return 0.

**3. v = 1 (no-op multiply):** Multiplying by 1 changes nothing. Your multiplier stays as-is, no issue. But it's a good sanity check that your loop terminates correctly.

**4. Multiple queries on the same index:** The multiplier for that index accumulates: if index 2 is hit by three queries with v=2, v=3, v=5, then `multiplier[2] = 30`. All mod $10^9+7$ as you go to avoid overflow.

**5. All elements XOR to 0:** Entirely valid output. Don't mistake 0 for "something broke."



## Lessons

**Deferred computation (lazy evaluation):** Instead of eagerly modifying values, track *what would happen* and apply it all at once. This is a recurring pattern in interval/range problems.

**Multiplier arrays:** Anytime a sequence of multiplicative operations targets the same elements, collapsing them into a single product per element saves repeated work.

**XOR loves deferred math:** XOR at the end only works here because the final value per index is fully independent of other indices. If the result required cross-element interaction, you'd have to be more careful.

**Modular arithmetic discipline:** Never let intermediate products grow unbounded. Every `*= v` should be immediately followed by `% MOD`.
