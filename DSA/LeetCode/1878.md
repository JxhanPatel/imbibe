# 1878. Get Biggest Three Rhombus Sums in a Grid 🟡
**Topics:** Array, Matrix, Prefix Sum, Enumeration



## Intuition

Imagine you have a piece of grid paper with numbers in each cell. You want to cut out diamond‑shaped frames (rotated squares) of different sizes, centered at any cell, and sum up only the numbers that lie exactly on the frame.  
A “rhombus sum” is just the sum of the numbers on the border of such a diamond. The smallest diamond is just a single cell (size 0). Larger diamonds are like picture frames – they have a hollow interior, and we only care about the border.

Our goal is to collect all possible diamond sums, keep only the distinct ones, and return the three largest in descending order.



## Approach

We can find every possible rhombus by choosing:
1. A **center** cell `(i, j)`
2. A **radius** `r` – the Manhattan distance from the center to any corner (top, bottom, left, right).  
   The corners are at `(i-r, j)`, `(i+r, j)`, `(i, j-r)`, `(i, j+r)`.

All cells on the border satisfy `|x-i| + |y-j| = r`. This gives us a neat way to enumerate the border: it consists of four diagonal line segments that meet at the corners.

For a given `(i, j)` and `r`, the maximum possible `r` is limited by the grid edges:  
`max_r = min(i, m-1-i, j, n-1-j)`.

Now, how do we compute the sum of the border efficiently?  
We can walk along the four sides (top‑left, top‑right, bottom‑left, bottom‑right). Each side is a diagonal line of length `r+1` that includes both its endpoints.  
If we sum all four sides, each corner will be counted **twice** (because it belongs to two sides). To fix that, we add the four sides together and then subtract the four corner values once.

The steps are:

- Loop over every cell as a possible center `(i, j)`.
- Add the single‑cell rhombus (radius 0) to our set of sums.
- For each radius `r` from 1 up to `max_r`:
  - Initialize `total = 0`
  - Add all cells of the top‑left side: `(i - r + t, j - t)` for `t = 0..r`
  - Add all cells of the top‑right side: `(i - r + t, j + t)` for `t = 0..r`
  - Add all cells of the bottom‑left side: `(i + r - t, j - t)` for `t = 0..r`
  - Add all cells of the bottom‑right side: `(i + r - t, j + t)` for `t = 0..r`
  - Subtract the four corner cells: `(i-r, j)`, `(i+r, j)`, `(i, j-r)`, `(i, j+r)`
  - Insert `total` into the set.
- Finally, sort the set in descending order and take the first three (or fewer) elements.

Because the grid is at most 50×50, the total number of (center, radius) pairs is under 125 000, and each rhombus has at most 200 border cells, so this approach runs easily within time limits.



## Example Walkthrough

Consider the grid from example 2:

```

```

Let’s examine a few rhombi:

- **Center (1,1) value 5, radius 1**  
  Corners: (0,1)=2, (2,1)=8, (1,0)=4, (1,2)=6  
  Border cells: exactly these four corners → sum = 2+8+4+6 = **20**

- **Center (2,2) value 9, radius 0** (single cell)  
  Sum = **9**

- **Center (2,1) value 8, radius 0**  
  Sum = **8**

- **Center (0,0) value 1, radius 0**  
  Sum = **1**

- **Center (0,0) radius 1** would require corners at (-1,0) and (0,-1) – out of bounds, so not allowed.

Collecting all distinct sums we get {20, 9, 8, 1, …}. Sorted descending: 20, 9, 8.  
That matches the example output `[20,9,8]`.



## Code

```python
class Solution:
    def getBiggestThree(self, grid: List[List[int]]) -> List[int]:

        m, n = len(grid), len(grid[0])
        distinct_sums = set()

        for i in range(m):
            for j in range(n):

                # size 0 rhombus: just the center cell
                distinct_sums.add(grid[i][j])


                # maximum possible radius before hitting a grid edge
                max_r = min(i, m - 1 - i, j, n - 1 - j)


                for r in range(1, max_r + 1):
                    total = 0


                    # top-left side
                    for t in range(r + 1):
                        x = i - r + t
                        y = j - t
                        total += grid[x][y]

                    # top-right side
                    for t in range(r + 1):
                        x = i - r + t
                        y = j + t
                        total += grid[x][y]

                    # bottom-left side
                    for t in range(r + 1):
                        x = i + r - t
                        y = j - t
                        total += grid[x][y]

                    # bottom-right side
                    for t in range(r + 1):
                        x = i + r - t
                        y = j + t
                        total += grid[x][y]



                    # subtract the four corners (each was counted twice, make sense)
                    total -= (grid[i - r][j] + grid[i + r][j] +
                            grid[i][j - r] + grid[i][j + r])

                    distinct_sums.add(total)

                

        # sort in descending order and take top 3 (or fewer maybe)
        sorted_sums = sorted(distinct_sums, reverse=True)
        return sorted_sums[:3]
            
```


## Complexity

- **Time:** $O(m \cdot n \cdot \min(m, n)^2)$  
  For each cell, we consider up to $\min(m, n)$ radii, and for each radius we loop over $4(r+1)$ border cells. The sum of $r$ over all radii per cell is $O(\min(m,n)^2)$.
- **Space:** $O(\text{number of distinct rhombus sums})$ – in the worst case this could be $O(m \cdot n \cdot \min(m,n))$, but we only store them temporarily in a set.



## Edge Cases

1. **Grid with only one row or one column**  
   Then `max_r` is always 0 for every cell, so only size‑0 rhombi (single cells) exist. The set contains all cell values, and we return the top three distinct values (or fewer).

2. **All grid values are identical**  
   Every rhombus sum will be the same number (e.g., `7` from example 3). The set will contain just one element, and we return `[7]`.

3. **Very large values (up to 10⁵) and many rhombi**  
   Sums can exceed 32‑bit integers, but Python handles big integers natively, so no overflow issues.



## Lessons

- **Manhattan distance** is a perfect tool to describe diamond shapes on a grid.
- When summing four overlapping sides, be mindful of double‑counting corners – a classic “inclusion‑exclusion” trick.
- Even with moderate constraints, brute‑force enumeration can be acceptable if we think clearly about the maximum possible number of iterations.
- Using a set automatically handles duplicates and simplifies the final selection of the top three distinct sums.
