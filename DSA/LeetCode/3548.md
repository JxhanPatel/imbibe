# 3548. Equal Sum Grid Partition II 🔴 
**Topics:** Array, Matrix, Hash Table, Prefix Sum

## Intuition

Imagine you're splitting a warehouse floor into two sections with a straight wall. You want both sides to store equal total inventory. Sometimes the split is naturally equal. Other times, you can move one item out of either section to balance the counts, but only if removing that item doesn't isolate part of the section (a corner pallet is fine to move, a middle pallet in a row blocks the path).

The question becomes: after each possible cut, can we find one cell whose value equals the imbalance, and whose position keeps its section connected if removed?

## Approach

Instead of checking every cell individually, we maintain two frequency counters (`top_freq` and `bot_freq`) that slide as we move the cut downward row by row. This lets us answer "does this value exist in this section?" in $O(1)$.

The connectivity rule for discarding a cell boils down to three cases per section:

- **Section is 1 row tall**: only the first or last cell in that row is safe to remove (endpoints of a single strip).
- **Section has 1 column wide** (n=1): every cell is a standalone strip, so only the top or bottom cell of the section matters.
- **Section is multiple rows tall and multiple columns wide**: any interior cell can be removed because the remaining rows keep the section fully connected. So we just check if `diff` exists anywhere in the section using the frequency counter.

For vertical cuts, we transpose the grid and reuse the exact same logic.

## Example Walkthrough

Using `grid = [[1,2],[3,4]]`, vertical cut (via transpose):

Transposed = `[[1,3],[2,4]]`

Cut after row 0: top=`[1,3]` sum=4, bot=`[2,4]` sum=6, diff=2. Bot is bigger, bot has 1 row, check endpoints: `g[1][0]=2`. diff==2, return `true`.

## Code

```python
from collections import Counter
from typing import List

class Solution:
    def canPartitionGrid(self, grid: List[List[int]]) -> bool:

        def check(g):
            m = len(g)
            n = len(g[0])

            if m < 2:
                return False  # need at least 2 rows to make a cut

            # start with everything in the bottom section
            bot_freq = Counter()
            for r in range(m):
                for c in range(n):
                    bot_freq[g[r][c]] += 1

            top_freq = Counter()
            row_sums = [sum(row) for row in g]
            tot = sum(row_sums)
            top_sum = 0

            for r in range(1, m):  # cut is between row r-1 and row r
                # move row r-1 from bottom to top
                for c in range(n):
                    v = g[r-1][c]
                    top_freq[v] += 1
                    bot_freq[v] -= 1
                    if bot_freq[v] == 0:
                        del bot_freq[v]

                top_sum += row_sums[r-1]
                bot_sum = tot - top_sum

                if top_sum == bot_sum:
                    return True

                diff = abs(top_sum - bot_sum)

                if top_sum > bot_sum:
                    # need to discard a cell from the top section
                    if r == 1:
                        # top is a single row, only endpoints are safe to remove
                        if diff == g[0][0] or diff == g[0][n-1]:
                            return True
                    elif n == 1:
                        # single column grid, top section is a vertical strip
                        # only top or bottom cell of the section is safe
                        if diff == g[0][0] or diff == g[r-1][0]:
                            return True
                    else:
                        # multi-row multi-col: any cell removal keeps connectivity
                        if diff in top_freq:
                            return True

                else:
                    # need to discard a cell from the bottom section
                    bot_rows = m - r
                    if bot_rows == 1:
                        # bottom is a single row, only endpoints are safe
                        if diff == g[r][0] or diff == g[r][n-1]:
                            return True
                    elif n == 1:
                        # single column grid, bottom section vertical strip
                        if diff == g[r][0] or diff == g[m-1][0]:
                            return True
                    else:
                        # multi-row multi-col: any cell removal keeps connectivity
                        if diff in bot_freq:
                            return True

            return False

        # check horizontal cuts directly
        if check(grid):
            return True

        # check vertical cuts by transposing and reusing the same logic
        transposed = [[grid[r][c] for r in range(len(grid))] for c in range(len(grid[0]))]
        return check(transposed)
```

## Complexity

**Time:** $O(m \cdot n)$ to build the counters, then $O(m \cdot n)$ total across all cut positions as each cell moves from `bot_freq` to `top_freq` exactly once. Transpose is another $O(m \cdot n)$. Overall: $O(m \cdot n)$.

**Space:** $O(m \cdot n)$ for the two frequency counters holding all cell values.

## Edge Cases

- **Single row or column**: `m < 2` guard returns `False` immediately for horizontal cuts. The transpose handles the equivalent for vertical cuts.
- **All cells equal**: The frequency counter still works correctly since it tracks counts, and the diff check will match the value if it exists in the right section.
- **n=1 grid**: Every row has one cell so removing it from a multi-row section would disconnect top and bottom. Only the endpoints of the section are safe, handled by the `n == 1` branch.
- **Cut where both sums are already equal**: Caught before even checking the discount logic, short-circuits immediately.

## Lessons

**1. Sliding frequency counter over a prefix.** Rather than recomputing "what values exist in the top section" from scratch at each cut, we maintain two counters and transfer one row at a time. This is the same mental model as a sliding window, just applied to set membership instead of a running total.

**2. Transpose to reuse logic.** Vertical and horizontal cuts are structurally identical problems. Rather than writing separate logic, transposing the grid lets you run the same function twice. Any time you see "do this for rows AND columns," transpose is usually cleaner than duplicating code.

**3. Connectivity by geometry, not graph search.** The problem sounds like it needs BFS to verify connectivity after removal. But geometric reasoning is enough: a cell in a multi-row, multi-column section can always be removed without disconnecting the section (the remaining rows form a connected rectangle). Only single-row or single-column sections need special endpoint checks. Recognizing when a hard graph problem collapses to a simple rule is a key competitive programming skill.
