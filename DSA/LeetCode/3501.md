# 3501. Maximize Active Sections After Trade II 🔴

**Topics:** Binary Search, Sparse Table, Range Maximum Query (RMQ), Prefix Processing, String

## Intuition

Imagine a long road where:

- `1` represents an **active section**.
- `0` represents an **inactive section**.

For every query, we are allowed to perform one trade that extends an existing active block by absorbing the zeros immediately beside it.

Instead of examining every character for every query, we first divide the road into **continuous blocks** of active and inactive sections.

Each active block already knows:

- how many zeros are on its left,
- how many zeros are on its right.

Now every query simply asks:

> "Among all active blocks that lie inside this interval, which one can gain the most additional active sections?"

To answer this quickly, we preprocess these gains using a **Sparse Table**, allowing maximum range queries in constant time.

This transforms an expensive per-query scan into a very fast lookup.


# Approach

The solution consists of four preprocessing steps.

## Step 1: Count existing active sections

```python
base_ones = s.count('1')
```

Every answer starts with the number of active sections already present.

The only thing each query changes is the number of extra zeros that can be converted.


## Step 2: Compress the string into runs

Instead of storing every character individually, consecutive identical characters are grouped together.

For example,

```text
001110011100
```

becomes

```text
(0,0,1)
(1,2,4)
(0,5,6)
(1,7,9)
(0,10,11)
```

Only the runs containing `'1'` are important.

For every active run we store:

- starting index
- ending index


## Step 3: Compute expandable zeros

For every active run we calculate

```text
left_zeros
```

Number of consecutive zeros immediately before the run.

and

```text
right_zeros
```

Number of consecutive zeros immediately after the run.

Example

```text
0001111000
```

For

```text
1111
```

we have

```text
left_zeros = 3
right_zeros = 3
```

If we could absorb both sides completely,

the gain would be

```text
3 + 3 = 6
```

So we precompute

```python
gain = left_zeros + right_zeros
```

for every active block.


## Step 4: Build a Sparse Table

Since every query asks

> Which active block has the largest gain?

we preprocess the gains using a Sparse Table.

This supports

```text
Maximum gain in any range
```

in

$$
O(1)
$$

time.

The preprocessing takes

$$
O(n \log n)
$$

only once.


# Processing Each Query

Each query provides

```text
[l, r]
```

representing a substring.

There are three possibilities.


## Case 1

No active block exists inside the query.

Example

```text
000000
```

Answer remains

```text
base_ones
```


## Case 2

Exactly one active block lies inside.

Example

```text
0001111000
```

Query

```text
[2,8]
```

Only part of the surrounding zeros are inside the interval.

So we compute

```python
min(left_zeros, available_left)
+
min(right_zeros, available_right)
```

Only the visible zeros can be absorbed.


## Case 3

Multiple active blocks lie inside.

Example

```text
000111000111000
```

Now there are three possibilities.

### Leftmost active block

Its left expansion may be clipped by the query boundary.

Its right expansion is complete.


### Rightmost active block

Its right expansion may be clipped.

Its left expansion is complete.


### Middle active blocks

These lie completely inside the query.

Their gain is unchanged.

Instead of checking them individually,

the Sparse Table immediately returns

```text
maximum gain
```

among all middle blocks.

Finally,

```python
extra = max(left_part, right_part, middle)
```

Answer becomes

```python
base_ones + extra
```


# Example Walkthrough

Suppose

```text
s = "0011100111000"
```

Active runs are

```text
Run 1 : 111
Run 2 : 111
```

Their surrounding zeros are

| Run | Left | Right | Gain |
|------|------|-------|------|
| 1 | 2 | 2 | 4 |
| 2 | 2 | 3 | 5 |

Now consider

```text
query = [1,11]
```

Both runs are inside.

The Sparse Table reports

```text
maximum gain = 5
```

If

```text
base_ones = 6
```

then

```text
answer = 6 + 5 = 11
```


# Code

```python
import bisect

class Solution:
    def maxActiveSectionsAfterTrade(self, s: str, queries: List[List[int]]) -> List[int]:
        
        n = len(s)
        base_ones = s.count('1')

        runs = []
        i=0

        while i<n:
            j=i
            while j<n and s[j]==s[i]:
                j += 1
            runs.append((s[i], i, j-1))
            i=j

        one_runs = [r for r in runs if r[0] == '1']
        cnt = len(one_runs)
        starts = [r[1] for r in one_runs]
        ends = [r[2] for r in one_runs]

        left_zeros = [0]*cnt
        right_zeros = [0]*cnt

        for idx in range(cnt):
            a, b = one_runs[idx][1], one_runs[idx][2]

            if idx == 0:
                prev_end = -1
            else:
                prev_end = one_runs[idx-1][2]
            left_zeros[idx] = a - prev_end-1

            if idx == cnt-1:
                next_start=n

            else:
                next_start = one_runs[idx+1][1]
            right_zeros[idx] = next_start-b-1


        gains = [left_zeros[k] + right_zeros[k] for k in range(cnt)]


        log_table = [0] * (cnt+1)
        for k in range(2, cnt+1):
            log_table[k] = log_table[k>>1]+1

        table = [gains[:]]
        k=1

        while cnt and (1 << k) <= cnt:
            prev_row = table[-1]
            half = 1 << (k-1)
            row_len = cnt - (1<<k) + 1
            table.append([max(prev_row[idx], prev_row[idx + half]) for idx in range(row_len)])
            k += 1

        def query_max(lo, hi):
            if lo>hi:
                return 0
            k = log_table[hi-lo + 1]
            return max(table[k][lo], table[k][hi - (1<<k) + 1])

        results=[]

        for l, r in queries:
            if cnt == 0:
                results.append(base_ones)
                continue

            left_idx = bisect.bisect_right(starts, l)
            right_idx = bisect.bisect_left(ends, r) - 1

            if left_idx > right_idx:
                extra=0

            elif left_idx == right_idx:
                k = left_idx
                extra = min(left_zeros[k], starts[k]-l) + min(right_zeros[k], r-ends[k])

            else:
                left_part = min(left_zeros[left_idx], starts[left_idx]-l) + right_zeros[left_idx]
                right_part = left_zeros[right_idx] + min(right_zeros[right_idx], r-ends[right_idx])
                middle = query_max(left_idx + 1, right_idx - 1)
                extra = max(left_part, right_part, middle)

            results.append(base_ones+extra)

        return results
```


# Complexity

Let

- $n$ be the length of the string.
- $m$ be the number of active runs.
- $q$ be the number of queries.

### Preprocessing

Run compression

$$
O(n)
$$

Building arrays

$$
O(m)
$$

Building Sparse Table

$$
O(m \log m)
$$


### Each Query

Binary searches

$$
O(\log m)
$$

Sparse Table lookup

$$
O(1)
$$

Therefore,

$$
O(\log m)
$$

per query.


### Overall

$$
O(n + m \log m + q \log m)
$$

Space complexity

$$
O(m \log m)
$$

for the Sparse Table.


# Edge Cases

### 1. No active sections

```text
s = "000000"
```

Every answer is `0` because there is no active block to expand.


### 2. Entire string is active

```text
s = "111111"
```

No surrounding zeros exist.

Every query simply returns the original number of active sections.


### 3. Query contains only part of a block

```text
00111000
```

The active block cannot absorb zeros outside the query boundaries.

Using `min()` ensures only visible zeros are counted.


### 4. Multiple active blocks

```text
000111000111000
```

The Sparse Table quickly finds the block with the maximum possible expansion without scanning every block.


# Lessons

- Compressing consecutive characters into runs often simplifies interval problems.
- Precomputing useful information about each run avoids repeated work during queries.
- Sparse Tables are ideal for static range maximum or minimum queries.
- Binary search can efficiently locate which runs intersect a query interval.
- Separating boundary cases from fully contained segments is a common optimization technique in interval-based algorithms.
