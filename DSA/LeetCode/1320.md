# 1320. Minimum Distance to Type a Word Using Two Fingers 🔴
**Topics:** Dynamic Programming, String





## Intuition

Imagine you are playing a piano with two fingers. At each note, you decide: should my left finger play this, or my right finger? The answer depends on where both fingers currently are, because moving a finger costs effort proportional to distance. You want to finish the whole song with the least total finger movement.

That is exactly this problem. The keyboard is your piano, and at every character in the word, you pick which finger types it. The other finger just rests wherever it was. Because each decision depends on both fingers' current positions, we need to track both simultaneously, which screams dynamic programming.






## Approach

We define a state as:

> `dp[i][f1][f2]` = minimum total distance to have typed `word[0..i]`, where finger1 is at letter `f1` and finger2 is at letter `f2`.

One important observation: at each step `i`, exactly one finger must be sitting on `word[i]`. So either `f1 == word[i]` or `f2 == word[i]`. This means we do not need to store both independently without constraint. We can simplify by only tracking the **other** finger (the one that did NOT just type the current character).

We use a 2D DP instead:

> `dp[i][j]` = min cost to type word up to index `i`, where the finger that did NOT just type `word[i]` is sitting at letter `j`. The letter `26` (a sentinel) means that finger has never been used.

At every step `i`, we consider two moves:
1. The same finger that typed `word[i-1]` now also types `word[i]`. Cost = `dist(word[i-1], word[i])`.
2. The other finger types `word[i]`. Cost = `dist(other_finger, word[i])` (which could be 0 if it has never moved, i.e., the first placement is free).

We take the minimum over both choices.







## Example Walkthrough: `"CAKE"`

Letters mapped to `(row, col)` where `idx = ord(c) - ord('A')`, row = `idx // 6`, col = `idx % 6`:
- C = (0, 2), A = (0, 0), K = (1, 4), E = (0, 4)

**Step 0: Type 'C'**
Both fingers are free. Placing a finger costs 0. The other finger is still at sentinel (26).

`dp[0][26] = 0`

**Step 1: Type 'A'**
- Option 1: same finger moves C → A, other finger stays at 26. Cost = dist(C, A) = |0-0| + |2-0| = 2. Other finger = 26.
- Option 2: new finger types A (free, first touch), other finger stays at C (= 2). Cost = 0. Other finger = 2.

`dp[1][26] = 2` (same finger moved)
`dp[1][2] = 0` (new finger took A, first finger at C = index 2)

**Step 2: Type 'K'**
For each previous state, try both move options again. Best we can find leads to cost 0 (new finger types K freely if it hasn't moved yet).

**Step 3: Type 'E'**
Costs accumulate. The minimum across all states at the end = **3**.








## Code

```python
def minimumDistance(word: str) -> int:
    # get (row, col) position of a letter on the keyboard
    def pos(c):
        idx = ord(c) - ord('A')
        return idx // 6, idx % 6

    # manhattan distance between two letters
    def dist(a, b):
        if a == 26:  # sentinel: finger never placed, first touch is free
            return 0
        r1, c1 = pos(chr(a + ord('A')))
        r2, c2 = pos(chr(b + ord('A')))
        return abs(r1 - r2) + abs(c1 - c2)

    n = len(word)
    INF = float('inf')

    # dp[j] = min cost when the "other" finger (the one that did NOT just type)
    # is sitting at letter j. j=26 means that finger is unplaced.
    dp = [INF] * 27
    dp[26] = 0  # start: both fingers free

    for i in range(n):
        cur = ord(word[i]) - ord('A')
        prev = ord(word[i - 1]) - ord('A') if i > 0 else 26
        new_dp = [INF] * 27

        for other in range(27):
            if dp[other] == INF:
                continue

            # option 1: same finger as last step types word[i]
            # other finger doesn't move, stays at 'other'
            move_cost = dist(prev, cur)
            new_dp[other] = min(new_dp[other], dp[other] + move_cost)

            # option 2: the other finger types word[i]
            # previous "typing finger" now becomes the idle one
            # so new "other" finger = prev
            move_cost = dist(other, cur)
            new_dp[prev] = min(new_dp[prev], dp[other] + move_cost)

        dp = new_dp

    return min(dp)
```







## Complexity

**Time:** $O(n \cdot 27)$ where $n$ is the length of the word. At each step we loop over 27 possible "other finger" positions.

**Space:** $O(27)$ since we only keep the current and previous DP row, effectively $O(1)$ relative to the alphabet.





## Edge Cases

1. **Word of length 2:** Only one transition. One finger types the first letter for free, the second letter either costs movement or a free second finger touch. The answer is often 0 if you assign smartly.

2. **All same character:** Like `"AAAA"`. Every move is finger1 typing the same position, cost = 0 every time. Answer = 0.

3. **Alternating between two distant letters:** Like `"AZAZAZ..."`. You can assign finger1 to 'A' and finger2 to 'Z' and never move either, so cost = 0 after the initial free placements. DP captures this naturally through the "other finger stays" option.

4. **Letters in the same row or column:** Distance is only the column or row difference, not a diagonal. Be careful not to use Euclidean distance.






## Lessons

- **When you have two "cursors" or agents and need to assign work between them, think 2D DP.** The classic pattern is: one of them always handles the current task, so you only need to track where the *other* one is.

- **Sentinel values are your friend.** Using `26` as "finger not placed yet" elegantly handles the free-first-touch rule without special-casing it everywhere.

- **At every step, only two transitions exist.** This is the key insight that keeps the complexity linear. You do not need to consider moving both fingers simultaneously, because only one finger types each character.

- **Reduce state whenever possible.** The naive state is `dp[i][f1][f2]` which is $O(n \cdot 26^2)$. Observing that one of the two fingers must always be on the current character collapses it to $O(n \cdot 27)$. That kind of observation is the difference between TLE and AC on Hard problems.
