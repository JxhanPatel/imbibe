# 3699. Number of ZigZag Arrays I 🔴

**Topics:** Dynamic Programming, Prefix Sum, Array

## Intuition

Imagine you are walking on stairs.

* If you take an **up step**, the next step must be **down**.
* If you take a **down step**, the next step must be **up**.

You are never allowed to go:

```text
up → up
```

because that would create a strictly increasing sequence of length 3.

Similarly, you are never allowed to go:

```text
down → down
```

because that would create a strictly decreasing sequence of length 3.

So the direction of movement must always alternate:

```text
up, down, up, down ...
```

or

```text
down, up, down, up ...
```

This observation turns the problem into a Dynamic Programming problem.

For every possible ending value, we only need to know:

1. How many arrays end here with the **last move increasing**.
2. How many arrays end here with the **last move decreasing**.

That is enough to build longer ZigZag arrays.


## Approach

Let

```text
m = r - l + 1
```

be the number of available values.

We define:

* `up[x]`

  Number of ZigZag arrays ending at value `x` where the last move was increasing.

* `down[x]`

  Number of ZigZag arrays ending at value `x` where the last move was decreasing.


### Initial State

For arrays of length `2`:

If the array ends at position `x`,

* all smaller values can come before it to make an increasing step.
* all larger values can come before it to make a decreasing step.

So:

```python
up[x] = x
down[x] = m - 1 - x
```

This is exactly what the code does:

```python
up = list(range(m))
down = list(range(m-1, -1, -1))
```

Example for:

```text
m = 5
values = [0,1,2,3,4]
```

```text
up   = [0,1,2,3,4]
down = [4,3,2,1,0]
```


### Transition

Suppose we want to end at value `y`.

#### Case 1: Last move is increasing

The previous value must be smaller.

Also, the previous move must have been decreasing.

Therefore:

```text
new_up[y]
=
sum of down[x]
for all x < y
```


#### Case 2: Last move is decreasing

The previous value must be larger.

Also, the previous move must have been increasing.

So:

$$
newDown[y]
$$

$$
\sum_{x>y} up[x]
$$


### Prefix Sums

Computing these sums directly would cost:

$$
O(m)
$$

for every state.

That becomes:

$$
O(nm^2)
$$

which is too slow.

Instead, we build prefix sums.

```python
psum_down[i+1] = psum_down[i] + down[i]
psum_up[i+1] = psum_up[i] + up[i]
```

Now,

```python
new_up[y] = psum_down[y]
```

because:

```text
psum_down[y]
=
down[0] + down[1] + ... + down[y-1]
```

Similarly,

```python
new_down[y]
=
total_up - psum_up[y+1]
```

which gives:

```text
up[y+1] + up[y+2] + ... + up[m-1]
```

Thus every transition becomes:

```text
O(1)
```

and the whole DP runs in:

```text
O(n × m)
```


## Example Walkthrough

### Input

```text
n = 3
l = 1
r = 3
```

Available values:

```text
[1,2,3]
```

So:

```text
m = 3
```


### Length = 2

```text
up   = [0,1,2]
down = [2,1,0]
```

Interpretation:

```text
up[2] = 2

Ending at value 3,
possible previous values:

1 -> 3
2 -> 3
```


### Build Length = 3

Prefix sums:

```text
psum_down = [0,2,3,3]

psum_up = [0,0,1,3]
```

Now:

```text
new_up[0] = 0
new_up[1] = 2
new_up[2] = 3
```

and

```text
new_down[0] = 3
new_down[1] = 2
new_down[2] = 0
```

Therefore:

```text
up   = [0,2,3]
down = [3,2,0]
```

Answer:

```text
sum(up) + sum(down)

= 5 + 5

= 10
```

which matches the expected output.


## Code

```python
class Solution:
    def zigZagArrays(self, n: int, l: int, r: int) -> int:

        m = r-l+1
        MOD = 1000000007

        if n==1:
            return m

        up = list(range(m))
        down = list(range(m-1, -1, -1))


        for _ in range(3, n+1):
            psum_down = [0] * (m+1)
            psum_up = [0] * (m+1)

            for i in range(m):
                psum_down[i+1] = (psum_down[i] + down[i])%MOD
                psum_up[i+1] = (psum_up[i] + up[i])%MOD


            total_up = psum_up[m]
            new_up = [psum_down[y] for y in range(m)]
            new_down = [(total_up - psum_up[y+1])%MOD for y in range(m)]
            up, down = new_up, new_down

        return (sum(up)+sum(down))%MOD
```


## Complexity

### Time Complexity

There are:

* `n` DP layers
* `m = r - l + 1` states per layer

Each layer is processed in linear time.

Therefore,

$$
O(n \times m)
$$

where

$$
m = r - l + 1
$$


### Space Complexity

We store:

* `up`
* `down`
* two prefix sum arrays

each of size `m`.

Hence,

$$
O(m)
$$


## Edge Cases

### 1. Minimum range

```text
n = 3
l = 4
r = 5
```

Only two values exist.

Possible arrays:

```text
[4,5,4]
[5,4,5]
```

Answer:

```text
2
```


### 2. Smallest valid length

```text
n = 3
l = 1
r = 2
```

Only alternating sequences are possible.

Answer:

```text
2
```


### 3. Large `n`

```text
n = 2000
l = 1
r = 2000
```

A naive DP with nested summations becomes:

$$
O(nm^2)
$$

which is too slow.

Using prefix sums reduces it to:

$$
O(nm)
$$

which easily fits the constraints.


## Lessons

* Sometimes a complicated condition can be rewritten as a condition on **differences** or **directions**.

* Tracking the **last state only** is often enough in Dynamic Programming.

* Prefix sums are extremely useful when DP transitions involve:

  ```text
  sum of values before i
  sum of values after i
  ```

* Reducing

  $O(nm^2)$

  to

  $O(nm)$

  using prefix sums is a very common optimization pattern in DP problems.
