# 3418. Maximum Amount of Money Robot Can Earn 🟡
**Topics:** Dynamic Programming, Matrix, Grid Traversal



## Intuition

Imagine you're driving from the top-left corner of a city to the bottom-right, and you can only turn right or go down. Along the way, some spots give you money, some spots have muggers who steal from you. But you have a special coupon: you can completely ignore two muggers on your entire trip. You want to maximize how much cash you have when you arrive.

The key insight is this: at every cell, what matters isn't just your position, but also how many "ignore a mugger" coupons you have left. That's exactly what drives our DP state.



## Approach

We use a 3D DP table where `dp[i][j][k]` means "the best coins we can have at cell `(i, j)` with `k` neutralizations still available."

At each cell, we look at where we came from (top or left), take the best, and then decide: if this cell has a robber and we still have neutralizations left, should we use one? We try both options and keep the max.

The transition at every cell looks like this:

1. Gather the best value arriving from above or from the left.
2. Add the current cell's coin value (could be negative, that's fine).
3. If the cell is a robber and `k > 0`, also consider arriving with `k-1` neutralizations but treating the cell as 0.

We fill the entire grid bottom-up, and the answer is the maximum across `dp[m-1][n-1][0]`, `dp[m-1][n-1][1]`, and `dp[m-1][n-1][2]`.

## Example Walkthrough

For `coins = [[0,1,-1],[1,-2,3],[2,-3,4]]`:

1. Start at `(0,0)`: `dp[0][0][2] = 0` (no moves yet, 2 neutralizations left)
2. Move right to `(0,1)`, coin=+1: `dp[0][1][2] = 1`
3. Move down to `(1,1)`, coin=-2: we USE a neutralization, so `dp[1][1][1] = 1` (not 1-2=-1)
4. Move right to `(1,2)`, coin=+3: `dp[1][2][1] = 4`
5. Move down to `(2,2)`, coin=+4: `dp[2][2][1] = 8`

Answer: **8**. See the diagram above for the full grid trace.

## Code

```python
class Solution:
    def maximumAmount(self, coins: List[List[int]]) -> int:

        m, n = len(coins), len(coins[0])

        NEG_INF = float("-inf")

        dp = [[[NEG_INF] * 3 for _ in range(n)] for _ in range(m)]

        dp[0][0][2] = coins[0][0]

        dp[0][0][1] = max(dp[0][0][1], 0)


        for j in range(1, n):

            for k in range(3):

                if dp[0][j-1][k] == NEG_INF:
                    continue

                val = dp[0][j-1][k] + coins[0][j]

                dp[0][j][k] = max(dp[0][j][k], val)

                if coins[0][j] < 0 and k > 0:
                    dp[0][j][k-1] = max(dp[0][j][k-1], dp[0][j-1][k])



        for i in range(1, m):
            
            for k in range(3):

                if dp[i-1][0][k] == NEG_INF:
                    continue

                val = dp[i-1][0][k] + coins[i][0]

                dp[i][0][k] = max(dp[i][0][k], val)

                if coins[i][0] < 0 and k > 0:
                    dp[i][0][k-1] = max(dp[i][0][k-1], dp[i-1][0][k])



        for i in range(1, m):

            for j in range(1, n):

                for k in range(3):

                    best_prev = max(
                        dp[i-1][j][k] if dp[i-1][j][k] != NEG_INF else NEG_INF,
                        dp[i][j-1][k] if dp[i][j-1][k] != NEG_INF else NEG_INF
                    )

                    if best_prev == NEG_INF:
                        continue


                    dp[i][j][k] = max(dp[i][j][k], best_prev + coins[i][j])


                    if coins[i][j] < 0 and k > 0:
                        dp[i][j][k-1] = max(dp[i][j][k-1], best_prev)


        return max(dp[m-1][n-1])
```

## Complexity

**Time:** $O(m \times n \times 3)$ which simplifies to $O(m \cdot n)$ since the 3rd dimension is constant. We visit every cell three times.

**Space:** $O(m \times n \times 3)$ for the DP table. You could optimize this to $O(n \times 3)$ by keeping only the previous row, but the full table is cleaner to understand.

## Edge Cases

Here are a few you should make sure your solution handles:

**All positive cells** (like Example 2): The robot never needs any neutralizations. The path cost is simply the sum along the cheapest route, and neutralizations are irrelevant. Make sure you don't accidentally assume they must be used.

**1x1 grid**: The start and destination are the same cell. The robot can collect the single coin (and even neutralize it if it's negative, though that just gives 0 vs a negative number). Your base case must handle this without trying to look left or up.

**More robbers than neutralizations**: The robot can only neutralize 2, so some negative cells WILL be paid. The path needs to route around the worst robbers, or eat the cost of the lesser ones. Don't assume all negatives get neutralized.

**All negative cells**: The total can still be negative. The problem explicitly says "the robot's total coins can be negative," so never cap your answer at 0.

**Robber at start `(0,0)`**: The robot can choose to immediately use one neutralization here. Make sure your base case handles `dp[0][0][1] = 0` in addition to `dp[0][0][2] = coins[0][0]`.

## Lessons

This problem teaches a beautifully general trick: **when you have a limited-use ability (skip, reroll, pass), add it as an extra dimension to your DP state.** Instead of tracking just WHERE you are, track WHERE you are AND HOW MANY uses you have left. This pattern shows up constantly in DP problems on grids, strings, and sequences.

The other key lesson is that our neutralization dimension only goes from 0 to 2, so the extra cost is constant. Don't let the 3D table intimidate you; it's really just a 2D grid with three "flavors" of each cell.

When you see "at most K special moves on a path," think: DP with a `k` dimension.
