# 2069. Walking Robot Simulation II 🟡

**Topics:** Math, Design, Simulation

## Intuition

Think of a racetrack. Cars don't follow a chaotic path around a field, they follow the **perimeter**. This robot is exactly the same. No matter how many steps you throw at it, it only ever moves along the outer edge of the grid.

So instead of simulating every single step (which would be painfully slow for large inputs), we ask: "where would the robot end up on this track after all these steps?" The answer is just `total_steps mod perimeter_length`. That's the whole trick.

The tricky bit is figuring out the direction. There's one annoying edge case: if you've taken exactly 0 total steps (you're back at the origin), your direction must be `"East"` not whatever the last move landed on. We'll handle that explicitly.

## Approach

We precompute every cell on the perimeter and its direction in order. Then for each `step(num)` call, we just advance a pointer along that list using modular arithmetic.

1. **Build the perimeter**: Walk the grid border in order (East along bottom, North up right side, West along top, South down left side), storing `(x, y, direction)` at each position.
2. **Track a pointer**: Keep a running index into this perimeter list.
3. **On each step call**: Move the pointer by `num`, wrapping around with `% perimeter_length`.
4. **On getPos / getDir**: Just read from the current pointer's stored data. Handle the origin-direction edge case.

## Example Walkthrough

Grid is 6x3. Perimeter length = `(6-1 + 3-1) * 2 = 14`.

| Call | Steps Taken | Pointer | Position | Direction |
|---|---|---|---|---|
| `Robot(6,3)` | 0 | 0 | (0,0) | East |
| `step(2)` | 2 | 2 | (2,0) | East |
| `step(2)` | 4 | 4 | (4,0) | East |
| `step(2)` | 6 | 6 | (5,1) then (5,2)... wait, pointer=6 | (5,1) North |
| `step(1)` | 1 more | 7 | (5,2) | North |
| `step(4)` | 4 more | 11 | (1,2) | West |

At pointer=11 on the "West" stretch: we've gone 11 steps. Bottom row has 5 steps (index 0-4), right col has 2 steps (index 5-6), so top row starts at index 7. Index 11 = 4 steps into the top row from right = x=5-4=1, y=2. That's `(1,2)` facing West.

## Code

```python
class Robot:

    TO_DIR = {
        0: "East",
        1: "North",
        2: "West",
        3: "South",
    }


    def __init__(self, width: int, height: int):
        self.moved = False
        self.idx = 0
        self.pos = []
        self.dirs = []
        pos_, dirs_ = self.pos, self.dirs


        # bottom row:- (0,0) to (width-1, 0), facing East
        for i in range(width):
            pos_.append((i, 0))
            dirs_.append(0)  # East

        # right col:- (width-1, 1) to (width-1, height-1), facin' North
        for i in range(1, height):
            pos_.append((width - 1, i))
            dirs_.append(1)  # North


        # top row:- (width-2, height-1) to (0, height-1), facing West
        # starts at width-2 because top-right corner already added above
        for i in range(width - 2, -1, -1):
            pos_.append((i, height - 1))
            dirs_.append(2)  # West


        # left col:- (0, height-2) to (0, 1), facing South
        # skips top-left (already added) and (0,0) which is index 0
        for i in range(height - 2, 0, -1):
            pos_.append((0, i))
            dirs_.append(3)  # South


        # (0,0) was assigned East on arrival, but after a full lap
        # the robot arrives there heading South — overwrite it
        dirs_[0] = 3


    def step(self, num: int) -> None:
        self.moved = True
        self.idx = (self.idx + num) % len(self.pos)

    def getPos(self) -> List[int]:
        '''return a copy so the internal tuple stays immutable'''
        return list(self.pos[self.idx])

    def getDir(self) -> str:
        # before any step, we're at (0,0) facing East by problem definition
        if not self.moved:
            return "East"
        return Robot.TO_DIR[self.dirs[self.idx]]
```

## Complexity

**Constructor:** $O(W + H)$ to build the perimeter list.

**step():** $O(1)$ — just modular arithmetic.

**getPos() / getDir():** $O(1)$ — direct array lookup.

**Space:** $O(W + H)$ for the perimeter list.

This beats the naive $O(num)$ simulation per step call by a massive margin, especially when `num` can be up to $10^5$ and there are $10^4$ calls.

## Edge Cases

**1. Already at (0,0) with direction "East"**
After enough steps you might lap back to index 0 in the perimeter. But index 0 stores direction `"East"` anyway since we started walking East. However, if `total_steps == 0` (no calls to `step` yet), we need to explicitly return `"East"`. Without the `total_steps` check, this naturally works, but it's good to be aware of.

**2. Large num values**
`num` can be up to $10^5$ and there can be $10^4$ calls. Without the `% perimeter_length` trick, you'd be doing $10^9$ iterations in the worst case. Always modulo.

**3. Minimum grid: 2x2**
Perimeter = `(1+1)*2 = 4` cells. The robot just loops around a tiny square. Make sure your range loops don't accidentally include corner cells twice (we stop at `width-1` and `height-1` in each side to avoid that).

**4. Very tall or wide grids (width=100, height=2)**
Perimeter is dominated by the horizontal stretches. The robot spends almost all its time going East or West. Works fine since the precomputed list handles any shape.

## Lessons

**1. Perimeter precomputation:** When a robot (or anything) is constrained to move along a border, precompute the border positions. This converts an $O(steps)$ simulation into $O(1)$ per query.

**2. Modular arithmetic for cycles:** Any time movement wraps around a fixed-length loop, `% loop_length` is your best friend. You don't care how many full laps were completed.

**3. Don't include corners twice:** When building the perimeter, each side's range excludes the last cell, because the next side picks it up as its first. Getting this wrong leads to off-by-one errors in position lookups.

**4. Watch out for "special" starting states:** The origin's direction after zero steps is a classic trap. The perimeter list "wants" to say East anyway, but explicit tracking of `total_steps` makes intent crystal clear and protects you from weird interview follow-ups.
