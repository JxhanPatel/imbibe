# 3130. Find All Possible Stable Binary Arrays II 🔴
Topics: Dynamic Programming, Prefix Sum Optimization

Hello there! This is a fantastic problem that looks intimidating at first glance but becomes incredibly satisfying once we crack the underlying pattern. We are going to use Dynamic Programming (DP) combined with a neat trick to keep our solution running fast.

## Intuition

Imagine you are a DJ creating a playlist for a party. You have exactly `zero` slow songs and `one` fast songs to play. However, you know the crowd will lose their vibe if you play more than `limit` slow songs in a row, or more than `limit` fast songs in a row.

If you are building this playlist one song at a time, how do you ensure it stays valid?
Whenever you add a new song (say, a slow song), you look at all the valid playlists you had built up to that point. You can add your new slow song to any of them. But there is a catch: what if adding this song pushes a streak of slow songs strictly over the `limit`?

Instead of looking backwards through every single song to check the streak, you can just calculate the total combinations and then **subtract the ones that just broke the rule**. A playlist only breaks the rule exactly right now if it was previously valid and ending in a fast song, followed by exactly `limit` slow songs.

## Approach

We will use Dynamic Programming to keep track of our valid playlists.

Let $dp[i][j][0]$ be the number of valid arrays using exactly $i$ zeros and $j$ ones, ending with a `0`.
Let $dp[i][j][1]$ be the number of valid arrays using exactly $i$ zeros and $j$ ones, ending with a `1`.

To figure out $dp[i][j][0]$ (adding a `0`), we add up all the valid arrays that had $i-1$ zeros and $j$ ones.
Formula: $dp[i-1][j][0] + dp[i-1][j][1]$

Then, we subtract the invalid combinations. An invalid combination happens if adding this `0` creates a streak of `limit + 1` zeros. This specifically occurs when we had a sequence ending in `1`, followed by exactly `limit` zeros. We can find exactly how many sequences look like that by checking our DP table from exactly `limit + 1` zeros ago: $dp[i - limit - 1][j][1]$.

We apply the exact same logic symmetrically for arrays ending in `1`.

## Example Walkthrough

Let us trace **Example 2**: `zero = 1`, `one = 2`, `limit = 1`.
We want to find all valid sequences using one `0` and two `1`s, with no 2 of the same number in a row.

1. **Base Cases**: We can make sequences of pure 0s or pure 1s up to the `limit` (which is 1).
* $dp[1][0][0] = 1$ (Sequence: `[0]`)
* $dp[0][1][1] = 1$ (Sequence: `[1]`)
* Note: $dp[0][2][1] = 0$ because 2 exceeds our limit of 1.


2. **Calculate for $i=1$ (one 0), $j=1$ (one 1)**:
* Ending in `0`: $dp[1][1][0] = dp[0][1][0] + dp[0][1][1] = 0 + 1 = 1$. (Sequence: `[1, 0]`)
* Ending in `1`: $dp[1][1][1] = dp[1][0][0] + dp[1][0][1] = 1 + 0 = 1$. (Sequence: `[0, 1]`)


3. **Calculate for $i=1$ (one 0), $j=2$ (two 1s)**:
* Ending in `0`: $dp[1][2][0] = dp[0][2][0] + dp[0][2][1] = 0 + 0 = 0$.
* Ending in `1`: $dp[1][2][1] = dp[1][1][0] + dp[1][1][1] = 1 + 1 = 2$.
Wait! Since $j$ (2) is greater than our limit (1), we must subtract the invalid sequences where we had a `0` followed by `limit` 1s. We subtract $dp[1][2 - 1 - 1][0]$, which is $dp[1][0][0]$ (value is 1).
Corrected ending in `1`: $2 - 1 = 1$. (Sequence: `[1, 0, 1]`. The invalid one we subtracted was `[0, 1, 1]`).


4. **Result**: Add the valid states for $i=1, j=2$: $0 + 1 = 1$. This perfectly matches the output!

## Code

```python
class Solution:
    def numberOfStableArrays(self, zero: int, one: int, limit: int) -> int:

        MOD = 10**9 + 7

        dp = [[[0, 0] for _ in range(one + 1)] for _ in range(zero + 1)]
        

        # fill sequences of pure 0s or pure 1s up to the limit
        for i in range(1, min(zero, limit) + 1):
            dp[i][0][0] = 1
            
        for j in range(1, min(one, limit) + 1):
            dp[0][j][1] = 1
            




        #DP table
        for i in range(1, zero + 1):
            for j in range(1, one + 1):
                
                # calculate sequences ending in 0
                dp[i][j][0] = (dp[i - 1][j][0] + dp[i - 1][j][1]) % MOD
                
                if i > limit:
                    dp[i][j][0] = (dp[i][j][0] - dp[i - limit - 1][j][1] + MOD) % MOD
                    


                # calculate sequences ending in 1
                dp[i][j][1] = (dp[i][j - 1][0] + dp[i][j - 1][1]) % MOD
                
                if j > limit:
                    dp[i][j][1] = (dp[i][j][1] - dp[i][j - limit - 1][0] + MOD) % MOD
                    
                    
        # The answer is the sum of valid sequences ending in 0 and ending in 1
        return (dp[zero][one][0] + dp[zero][one][1]) % MOD

```

## Complexity

* **Time Complexity**: $O(\text{zero} \times \text{one})$. We iterate through a nested loop up to `zero` and `one`. The operations inside the loop run in constant time $O(1)$. Given the constraints of $1000$, this will execute around $10^6$ operations, which easily passes.
* **Space/Memory Complexity**: $O(\text{zero} \times \text{one})$. We store a 3D array of size `zero` $\times$ `one` $\times 2$.

## Edge Cases

1. **Impossible combinations**: If `zero` is massive (e.g., 100) and `one` is tiny (e.g., 1) with a small limit (e.g., 2), the math will naturally cascade to 0 because every addition will be immediately balanced by the limit subtraction.
2. **`limit` is larger than both `zero` and `one**`: The subtraction condition `i > limit` and `j > limit` will never trigger, naturally degrading the problem into standard combinatorics (finding all simple permutations).
3. **`limit` is exactly 1**: The algorithm perfectly handles this strict alternating pattern without requiring special helper logic.

## Lessons

The biggest takeaway here is the **"Inclusion-Exclusion" DP pattern**. When you are asked to count sequences with a maximum consecutive constraint, do not use an inner loop to check lengths. Instead, optimistically add all previous states and subtract only the boundary state that breaks the rule. This trick drops the time complexity by an entire order of magnitude!
